# Todica（トディカ）

由来: `Todo`（ToDo）と `Today`（今日）を掛けた "Tod-" に, 接尾辞 `-ica`（ラテン語由来. "〜の領域 / 世界"）を付けた造語. 

---

## 1. Project Summary

個人向けに機能を絞った, **"今日やることにフォーカスした ToDo リスト"**（オープンソース）. 
自分の課題を解決するために作り, 自分が使い続けられることを最優先とする. 
最終的に Google Play Store へ公開することをゴールとし, 公開そのものを目的とする（他者の利用数は問わない）. 

プロジェクト管理ソフトの "個人には機能過多" と, 汎用 ToDo の "タスク単体では文脈不足" の“間”を埋める. 
プロジェクトという文脈を保ちつつ, タスクは最小手数で作ることができる. 
今日のタスクから1つを選んで今はそれだけに集中させる. 
プロジェクト管理ソフト（Asana 等）とは競合せず補完する. 

## 2. Purpose

個人が先延ばしや脇道に逸れることなく, "今日やること" を毎日確実に終わらせていけるよう支える. 
そのために, タスク化の手間と管理の負担を限界まで減らし, 進行中に設定できるタスクは "今やる1つ" だけにする. 

## 3. Design Principles（設計原則）

差別化の本質は "機能の多さ" ではなく "何を作らないか" にある. 
それを次の2軸に集約し, 最上位の設計判断基準とする. 

### 中核となる 2 つの軸

- CORE-1: 意見の強い単一ワークフロー. 
  - カスタマイズ余地を削り, "プロジェクト作成 → タスク（既定期限=今日）を最小手数で足す → 今日のタスクから1つ選んで集中 → 完了で自動的に次へ" の一本道を強制する. 
  - 設定で迷う時間をゼロにする. 
- CORE-2: 完全な個人専用. 
  - マルチユーザー概念を一切持たない. 
  - 担当者・権限・共有・コラボを排し, データモデルと UI を徹底的に単純化する. 

### 設計の判断基準（Design North Star）

新しい機能・仕様を検討するときは, 必ず次を問う. 

1. それは "設定・選択肢・汎用性" を増やすか? → 増やすなら原則として却下（CORE-1 に反する）. 
2. それは "複数ユーザー / 協働" を前提とするか? → 前提とするなら却下（CORE-2 に反する）. 
3. それは "今日にフォーカスして消化する" 唯一のワークフローを, より迷わず・速くするか? → するなら採用候補. 

## 4. Target User

"今やるべきこと" を, 自分専用の手元の場で気軽に整理したい人. 
具体的には次のような人を想定する. 

- チーム開発に参加するエンジニア. 
  - Asana / Jira 等でチームのチケットは管理しているが, 起票前の検討や細かい段取りを手元で回したい. 
- 複数の個人プロジェクトを並行する人. 
  - 副業・趣味・学習などを抱え, 重い PM ツールを使うほどではないが今日やることを整理したい. 
- やることが散らかって手が止まる人. 
  - 頭の中や複数ツールに散らばったタスクを一箇所に集め, 今やるべきことを明確にしたい. 
- 新しいことを習慣化したい人. 
  - 連続記録にとらわれないで, 間欠的でもいいから続けたい人. 

## 5. Problem

"やるべきことが終わらない" という問題の核心は, 自分が "今" やるべきことに集中できないことにある. 
やるべきことが頭の中や複数のツールに散らばり, 今この瞬間に着手すべき一手を見失いやすい. 
作業中に脇道へ逸れると, 本来やるべきだった一手に戻ってこられないこともある. 

既存の手段にはそれぞれ次の弱点がある. 

- プロジェクト管理ツールの場合
  - テキトーに登録したタスクがポートフォリオなどに影響するため, 気軽にチケットを発行しにくい. 
  - すぐに終わる作業なのにいちいち期限や担当者を入力する必要がある. 
  - チケット作成そのものに必要な作業を登録しづらい. 
  - プロジェクトが遅れてくるとあれもこれもやろうとしてしまう. 
- ToDo リストは期限が緩い. 
  - 期限を決めなくてもよく, はるか先の未来まで設定できてしまう. 
  - どのタスクをやるか決める段階で時間がかかる. 
  - 未来の作業がちらつき, 今の作業に集中できない. 

必要なのは, 粒度にこだわらず "今やる作業" をとりあえず気軽・高速に書き留められ, かつ "今やる1つ" に立ち返れる場所. 
すぐに登録でき, 後から自由に追加・削除でき, 今集中すべき1つを見失わないことが重要. 

## 6. Value Proposition

- 気軽・高速にタスク化できる. 
  - 粒度や体裁を気にせず, 今やる作業をとりあえず書き留められる. 
  - 所属プロジェクトが文脈を補うので, タスク名は簡潔でよい. 
  - 後から自由に追加・削除できる. 
- 今やる1つに集中できる. 
  - 今日のタスクから1つを選ぶと, それだけが大きく表示される. 
  - 脇道に逸れても, 立ち返れる場所になる. 
  - 完了すると自動で次のタスクに進む. 
- 設定で迷わない. 
  - 期限は今日 / 明日だけで, 未来の作業に気を取られない. 
  - 唯一の決められたワークフローを強制し, カスタマイズの判断コストをゼロにする（汎用 ToDo との最大の差）. 
  - タスクどうしに関連性はなく, 担当者はユーザーだけ. 
- 完全な個人専用. 
  - マルチユーザー概念を排し, 個人の使い方に徹底最適化したシンプルな操作性. 

## 7. Use Cases

### UC-001: 今日のループを回す（入口・フォーカス）

**Goal:** 今日やることを, "今やる1つ" に集中して消化する. 

**Flow:**
1. アプリを開くと, 今日のタスク（全プロジェクト横断 + プロジェクト外 + ルーティン）が優先度順に出る. 
2. 1つを "現在のタスク" に選ぶと, それだけが大きく表示される（選択中=進行中・他=未着手）. 
3. 終わったら完了アクションで消化する. 自動で次（優先度最上位）が現在のタスクになる. 
4. 今日できないものは期限を明日にずらす. 
5. 脇道に逸れたら, 現在のタスク画面に戻って立ち返る. 

**Expected Outcome:** 計画に悩まず, 今やる1つに集中して上から消化できる. 

### UC-002: タスクを気軽に起票する

**Goal:** 思いついた作業を, 粒度を気にせずすぐ記録する. 

**Flow:**
1. 今やる作業を最小手数で書き留める（プロジェクトに紐づけても, 単発でもよい）. 
2. 期限は既定で今日（今日 / 明日のみ変更可）. 
3. 後から優先度を変えたり, 不要なら削除したりできる. 

**Expected Outcome:** 粒度や分類に悩まず, とりあえず ToDo にして手を動かせる. 

### UC-003: プロジェクトで進める

**Goal:** 関連するタスクをプロジェクトにまとめて消化する. 

**Flow:**
1. プロジェクトを作る. 
2. タスクを足して, 日次ループ（UC-001）で消化する. 
3. 不要になったプロジェクトは削除する（ゴミ箱へ. 同日内は復元可）. 

**Expected Outcome:** 関連タスクをまとめて扱え, 不要になれば手軽に片付けられる. 

### UC-004: ルーティンを自動でタスク化する

**Goal:** 繰り返す作業を, 毎回手で登録せずに今日のタスクにする. 

**Flow:**
1. ルーティンを一度だけ定義する（生成する曜日・既定優先度を指定）. 
2. 指定曜日に, 自動で "今日のタスク" として生成される. 
3. 完了したら消化される. 
4. その日に実施しなくても, 日付の切替で自動削除される（翌日に持ち越さない）. 

**Expected Outcome:** 繰り返しの登録の手間がなくなり, 毎日リセットされた状態で向き合える. 

### UC-005: プロジェクト管理ツールと併用する

**Goal:** PM ツール（Asana 等）はそのまま使いつつ, 自分の "今やること" だけを手元で回す. 

**Flow:**
1. PM ツールに起票する前の検討や段取りを, 本ツールに自分用のタスクとして気軽に書く（粒度は任意）. 
2. PM ツール側のチケットのうち, 今日やるぶんだけを手で本ツールに書き写す（簡潔でよい. 終わったら消す）. 
3. 1つのチケットが複数手順（例: 実装 → 報告）を含むなら, 別タスクに分けて今日やるぶんだけ進める. 

**Expected Outcome:** PM ツールの全体像に煩わされず, 今日の自分の作業に集中できる. 両者は独立しており, 自動連携はしない. 

## 8. Scope

### In Scope

- プロジェクトの管理（複数プロジェクト. 役割は "タスクに文脈を与える". 不要になれば削除する）
- "今日" ビュー（全プロジェクト横断・期限=今日のタスクを優先度順に表示）= アプリの入口
- 現在のタスク（フォーカス）ビュー（今日のタスクから1つ選んで大きく表示. 完了で自動的に次へ繰り上げ）
- プロジェクト → タスクの 2 階層（タスクが最小単位）
- タスク管理（期限つき. 既定=今日, 設定は今日 / 明日のみ）
- タスクの優先度（3 段階: 最優先 / 普通 / 後回し）
- 完了アクション（ゴミ箱へ移動＋今日の完了数 +1）
- 削除アクション（ゴミ箱へ移動. カウントには含めない）
- "今日の完了タスク数" のカウント表示（日次リセット. リセット時刻はユーザー設定可）
- 未完了タスクの自動繰越（日跨ぎで期限=今日・優先度=最優先. ルーティン由来を除く）
- ゴミ箱（削除はすべてゴミ箱経由・同日内は復元可. リセット時に自動完全削除＋手動 "空にする" ）
- ルーティン機能（独立概念. 指定曜日に自動生成, 完了 / 期限切れで削除, 履歴なし）
- プロジェクト外タスク（単独タスク）
- 単一ユーザーでの利用

### Out of Scope

- 進捗管理 / 進捗率・進捗バー
- プロジェクトのアーカイブ（区切りがついたら削除する. 残して整理する概念は持たない）
- サブタスク / サブプロジェクト / ネスト（タスクを最小単位に固定し, 思考コストを下げる）
- 検索機能（検索が要るほど膨大＝既に管理不能, という思想 / 量を増やさない設計圧）
- "保留" / ユーザーが直接設定するステータス, カスタムフィールド, タグ・ラベル
- ビュー切替（カンバン / ガント / カレンダー / テーブル）, 保存フィルタ, ユーザー定義ソート
- 開始日, 3 日以降の期限指定, （通常タスクの）繰り返し, リマインダー通知
- ルーティンの実施履歴 / ストリーク
- 成果物そのもののファイル管理・保存
- 外部連携 / 自動化ルール / カスタマイズ設定画面
- 複数ユーザー / チーム機能（担当者アサイン, 権限管理, 共有など）
- 共有 SaaS としてのホスティング提供

## 9. Success Criteria

- SC-001: 本人が, 日々の "やること" を本ツールに集約し, 1 か月以上日常的に使い続けられる. 

## 10. Assumptions（未検証仮説）

- ASM-001: 尖った設計（機能を絞り, 今日 / 明日のみ・1つに集中）が, 窮屈さに転じず "今に集中して今日ぶんを終わらせる" 効果を生む. 
- ASM-002: 実施履歴・ストリークがなくても, 日々の利用継続は損なわれない. 
- ASM-003: コードを大きく作り直さずに Android アプリ化でき, Google Play Store 公開まで現実的なコストで到達できる. 

## 11. Constraints

- CONSTRAINT-001: 開発・運用は個人1人で行う（時間・リソースが限られる）. 
- CONSTRAINT-002: コストは個人負担とし, 高額な有料サービス・ライセンスを前提にしない. 
- CONSTRAINT-003: 公開先である Google Play Store の規約・審査に従う. 

## 12. Feasibility Summary

### Technical Feasibility

- 機能は小規模な CRUD と日次処理が中心で, 特殊な技術や難所がない. 
- Web から Android への展開とストア公開は, 確立した手法で実現できる（前例が豊富）. 

### Operational Feasibility

- 個人規模で完結し, 大規模インフラを必要としない. 

## 13. Key Risks

- RISK-001: 自分の日々の道具として定着せず, 使われなくなる. 
  - 影響: 大. SC-001 未達でプロジェクトの存在意義が失われる. 
  - 対策: 最小機能で早期にドッグフーディングし, 毎日使い続けられるかを検証する. 
- RISK-002: 同一コードの Android 化・Google Play Store 公開が想定以上に難航し, 公開ゴールに届かない（ASM-003 が外れる）. 
  - 影響: 中. 公開という目的が未達となる. 
  - 対策: 早い段階で Android ビルド・公開フローを小さく試し, 流用コストと公開手順を確認する. 
- RISK-003: 個人1人の開発が継続できず, 完成・公開・保守が止まる（CONSTRAINT-001）. 
  - 影響: 中. 開発・運用が止まり, ツールが陳腐化する. 
  - 対策: スコープを個人用途に絞り機能を最小限に保つ. まず自分が使える最小形を優先する.
