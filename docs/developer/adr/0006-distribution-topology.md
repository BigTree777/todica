## ADR-0006: 配布構成の再定義（Web=自前サーバ接続 / Android=ローカル or サーバ選択可）

> Todica の配布構成と, クライアント・サーバの分担を定義し直す. 以前は「クライアント単体で完結」と整理していたが, ユーザーとの確認によって構成前提が変わったため, 本 ADR で新しい前提を明文化し, 後続の技術選定 ADR の土台にする.

- 状態: 承認（2026-06-07. 本 ADR の承認により ADR-0003 / 0004 / 0005 の廃止が確定）
- 日付: 2026-06-05
- 決定者: プロジェクトオーナー（個人開発）

## 背景 / 状況

- 以前の architecture / ADR-0003 / 0004 / 0005 は「Todica はクライアント単体で完結する SPA / PWA. サーバ運用ゼロ. Android は同じ Web 実装を WebView ラップで配布」を前提に組まれていた.
- ユーザーとの確認の結果, 実際の構成は次のとおりであることが確認された.
  - **Web アプリ → 自前サーバに接続. Web 上のデータは自前サーバが保持する**.
  - **Android アプリ → ローカル保持 *または* 自前サーバ接続のいずれかを選べる**.
    - ローカル単独モード: Android 端末内にデータを持ち, Web とは独立して使う.
    - サーバ接続モード: Web と同じ自前サーバに接続し, Android と Web が同じデータを見る.
- ここでいう「自前サーバ」は **ユーザー本人が運用する単一インスタンス** を指す. マルチテナントの共有 SaaS にはしない（project.md §8 Out of Scope "共有 SaaS としてのホスティング提供" を引き続き遵守）.
- 単一ユーザー（本人）のためのバックエンドであり, project.md §3 CORE-2「マルチユーザー概念を一切持たない」と整合する.

## 検討した選択肢

| 選択肢 | 利点 | 欠点 |
| --- | --- | --- |
| A: Web=サーバ接続必須 / Android=ローカル or サーバ選択可（本案） | 端末をまたいで同一データを扱える（Web ↔ Android 間共有）. Android はオフライン・スタンドアロンでも使える. Web ブラウザを持ち歩く必要がなくなる. | サーバ運用の手間とコストが発生（自前 VPS / 自宅サーバ / 無料枠 PaaS 等が候補）. Android の 2 モード切替の実装コストが増える. |
| B: Web も Android も常にサーバ接続必須（ローカルモードなし） | 構成が一本化されて実装がシンプル. データの正本がサーバに一意に存在する. | Android がサーバ無しでは動かないので, 個人で Android だけ使いたいユーザー（自分自身）が常にサーバを立てる必要がある. 個人運用コストが上がる. |
| C: Web も Android も常にローカル保持（サーバなし. ADR-0003 〜 0005 の旧前提） | サーバ運用ゼロ. シンプル. | Web 利用時にデータが「そのブラウザのオリジン領域」に閉じ, 端末をまたぐと別データ扱いになる. ユーザー意図（Web のデータは自前サーバが保持する）と合わない. |

## 決定

**A を採用する**. すなわち次の構成を Todica の前提とする.

- Web クライアントは **自前サーバへの接続を前提** とする. ブラウザ単独でデータを完結させない（オフライン時の挙動は ADR-0008 / feature spec で扱う. キャッシュによる読み取り表示は妨げない）.
- Android クライアントは **ローカルモード / サーバモード のいずれかを起動時または設定で選べる**. 一度モードを選んだ後の切替方法（再インストール時のみ / いつでも切替可能 / 切替時にデータ移行する など）は ADR-0009 で扱う.
- 自前サーバは **本人運用の単一インスタンス**. 認証はマルチテナント化せず, 単一アカウントで十分（詳細は ADR-0010）.
- 共有 SaaS としての公開 API ・マルチテナントホスティングは提供しない（project.md §8 Out of Scope を遵守）.

## 結果 / 影響

- 良い影響
  - 「Web と Android が同じデータを見られる」「Android はサーバ無しでも単独で動く」というユーザー意図がそのまま満たせる.
  - 単一ユーザー前提を維持したまま, サーバ運用は「本人の本人による本人のためのインスタンス」に閉じる. project.md §8 Out of Scope "共有 SaaS としてのホスティング提供" の規律を破らない.
- トレードオフ / 注意点
  - サーバを 1 つ運用するコストが発生する. 個人負担コスト前提（NFR-032, CONSTRAINT-002）と整合する範囲で選定する必要がある（ADR-0007）.
  - Android の 2 モード対応により, クライアント側の抽象（ローカルストレージ実装と API クライアント実装を同じインターフェースで扱う層）が必要になる. ADR-0009 と architecture/module-boundaries.md で扱う.
  - 「サーバが落ちた / 通信できない」ときの Web の振る舞いは別途設計が要る（読み取りキャッシュ・楽観 UI の方針は ADR-0008 / feature spec）.
- requirements.md NFR-021（「データの場所やアクセスする端末を制限しない」）と整合する.
- requirements.md NFR-031（「同一コードベースから Android アプリとして配布できる構成」）については, 「同一コードベース」を厳密に 1 つに保つかどうかは ADR-0008 / 0009 で再評価する（Capacitor で 1 コードに寄せる案・Android はネイティブ別実装にする案 などが候補）.
