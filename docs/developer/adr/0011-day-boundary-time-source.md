## ADR-0011: 境界時刻処理の時刻基準（サーバ時刻 vs クライアント時刻）

> 「今日」と「翌日」を切り替える境界時刻処理（FR-041 〜 FR-043, FR-051, FR-062）について, クライアント・サーバ構成の前提でどちらの時刻を正本にするかを決める.

- 状態: 承認（2026-06-07）
- 日付: 2026-06-05
- 決定者: プロジェクトオーナー（個人開発）

## 背景 / 状況

- 構成は ADR-0006 のとおり. Web / Android（サーバモード）はサーバ接続. Android（ローカルモード）は端末内完結.
- 境界時刻 + タイムゾーンはユーザー設定（FR-042, NFR-020）. 単一ユーザーなのでタイムゾーンは固定で良いが, 設定として持つ.
- リセット処理は冪等であること（NFR-020）. 二重繰越・二重リセットを起こさない.
- 「アプリが開かれていないとき」もリセットは将来のいつか実行される必要がある（FR-043）. 起動時の追いつき処理がある.

## 検討した選択肢

| 選択肢 | 利点 | 欠点 |
| --- | --- | --- |
| T1: サーバ時刻を正本とする. サーバが境界時刻にバッチ的にリセット処理を実行（または次のクライアントアクセス時に lazy に実行）. クライアントは表示・通知のみ. | データの正本がサーバ側にある場合（Web 利用時 / Android サーバモード時）と整合. 端末時刻のずれを気にしなくて良い. 「最後にリセットを実行した境界時刻」もサーバ側で一意に管理できる. | Android ローカルモードはサーバを持たないため, この方式単独では対応できない. ローカルモードでは端末時刻に頼らざるを得ない. |
| T2: クライアント時刻を正本とする. 各クライアントがリセット処理をオーケストレートする. | Android ローカルモードはそのまま対応できる. クライアントの時刻はユーザーに分かりやすい. | サーバモードでは複数クライアント（Web + Android）が同じデータを共有する場合, それぞれが独立にリセットを起動すると二重実行のリスクが出る. 冪等性の担保がより難しくなる. |
| T3: ハイブリッド. サーバモード時はサーバ時刻, ローカルモード時はクライアント時刻. リセット処理自体は同じドメインロジックで実装する. | 各モードで自然な時刻基準を選べる. ローカルモードは ADR-0004 旧版と同等の挙動になる. | 実装上, モードによって時刻ソースを差し替える層が必要. 仕様としてもユーザーに「Web で見たときとローカルモードで見たときは時刻基準が違う可能性がある」と説明する必要がある. |

## 決定

**T3（ハイブリッド）を採用する.**

- サーバモード時（Web / Android サーバモード）: **サーバ時刻が正本**.
  - リセット処理はサーバ側で実行する. 「lazy 起動（次のクライアントアクセス時に未実行の境界時刻があれば実行）」を基本とし, 必要なら cron / 定期実行で補強する.
  - クライアントは「サーバから取得した状態」を表示する. クライアント側で再計算しない.
- ローカルモード時（Android ローカルモード）: **クライアント時刻が正本**.
  - リセット処理は Android アプリ内で実行する. 「起動時 / フォアグラウンド境界時刻到達時」の両経路（旧 architecture/overview.md と同じ方針）を用いる.
- 共通: 境界時刻 + タイムゾーンの設定は両モードで同じ意味を持つ. ドメイン層の Clock 抽象（architecture/module-boundaries.md）は同じインターフェースで実装される.

## 結果 / 影響

- 良い影響
  - サーバモードでは複数クライアントが同居しても二重リセットを起こさない（サーバが一意の真実源）.
  - ローカルモードはサーバが要らない. オフラインで完結する.
- トレードオフ / 注意点
  - 「ローカルモードで使っていた Android を後でサーバモードに切替えたい」というケースは, リセットの履歴整合性を別途設計する必要がある（feature spec で扱う / または ADR-0009 のモード切替方針 案 1 の「再インストール扱い」で回避）.
  - サーバ・クライアントの時計ずれは, クライアントが直接時刻計算しない（サーバから状態を取得する）方針で吸収する.
