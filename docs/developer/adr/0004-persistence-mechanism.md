## ADR-0004: 永続化方式の選定（ブラウザローカル前提）

> 本 ADR は「単一ユーザー・端末ローカル前提・端末間同期なし」を前提として IndexedDB を推奨案にしていたが, 構成前提が「Web=自前サーバ接続必須 / Android=ローカル or サーバ選択可」に修正されたため Superseded とする. 永続化方式は以下の 2 ADR で分けて扱う.
>
> - サーバ側永続化（Web の保存先 / Android のサーバモード時の保存先） → [ADR-0007](0007-server-tech-stack.md)
> - Android のローカルモード時の永続化 → [ADR-0009](0009-android-client-tech-stack.md)

- 状態: 廃止（→ [ADR-0007](0007-server-tech-stack.md), [ADR-0009](0009-android-client-tech-stack.md)）
- 日付: 2026-06-05（廃止: 2026-06-05）
- 決定者: プロジェクトオーナー（個人開発）

## 背景 / 状況

- Todica は単一ユーザーでの利用に閉じる（NFR-021, NFR-002）. サーバ側 DB を置かない.
- 端末間同期・共有 SaaS としてのホスティングは行わない（OOS-012）.
- データはタスク・プロジェクト・ルーティン・ゴミ箱・カウントなど, 個人規模では極めて小さい（〜数 MB 程度を想定）.
- 「リセット処理が中断・再起動を挟んでも二重繰越を起こさない」整合性要件がある（NFR-020, FR-043, FR-051）. つまりトランザクション的な書き込みが扱えると望ましい.
- 技術スタックは Web 先行・Android は WebView ラップを想定する（ADR-0003）. このため永続化はブラウザの API で完結する必要がある.

## 検討した選択肢

| 選択肢 | 利点 | 欠点 |
| --- | --- | --- |
| A: IndexedDB（軽量ラッパー: Dexie 等の利用は実装側判断） | ブラウザ標準. 構造化データ・インデックス・トランザクションを扱える. WebView 経由の Android でもそのまま動く. 個人規模のデータ量は余裕で収まる. | 生 API は冗長で扱いにくく, 何らかの薄いラッパーが事実上必要. |
| B: localStorage に JSON を全件直列化 | 実装が極めて単純. 1 ファイル相当として扱える. | 同期 API のため UI スレッドをブロックしやすい. 5〜10 MB 程度の容量制限. トランザクションがなく, リセット中断時の整合性確保（NFR-020）が自力での実装になる. |
| C: SQLite を WebAssembly 経由（sql.js / wa-sqlite 等）でブラウザ内に持つ | SQL でクエリでき, スキーマ/制約が明確. Android 側でネイティブ SQLite に乗せ替える際の概念が一致する. | ランタイムバンドルが数百 KB 以上重くなる. ファイル永続化は結局 IndexedDB か Origin Private File System を介す必要があり, 層が増える. 個人開発の保守コストが上がる. |

## 決定

**現時点では A（IndexedDB を主たる永続化, 必要に応じて軽量ラッパーで扱う）を推奨案として提示し, B / C を代替案として併記する. 確定はユーザー判断を仰ぐ.**

- 推奨理由
  - データ量・整合性要件・WebView 互換性のいずれも IndexedDB で十分満たせる.
  - トランザクションを使えるため, リセット処理（FR-043, NFR-020）を一括 commit で安全に書ける.
  - Android 化（Capacitor 経由）でもブラウザの API 互換性に乗るため, 永続化の作り直しが要らない（NFR-031）.
- 代替案を採る条件
  - 「データ件数が極小で同期 API でも問題ない」と割り切れるなら B（localStorage）も成立する. ただし整合性は自力で担保する必要があるため, MVP 以降の保守でコストが効いてくる.
  - 後年, クエリの複雑化や端末間同期が要件に上がった場合は C（SQLite/WASM）を再評価する.

## 結果 / 影響

- 良い影響
  - 単一ユーザー・ローカル前提に最小限の実装で対応でき, サーバ運用ゼロを維持できる（NFR-032, OOS-012）.
  - リセット処理・自動繰越が atomic に書けるため, NFR-020 を構造的に満たしやすい.
- トレードオフ / 注意点
  - IndexedDB の生 API は使いにくい. ラッパーの選定（Dexie 等）は feature spec / implementer 側で決める.
  - エクスポート / インポートが必要になった場合の手段は別途仕様化する（本 ADR の範囲外）.
  - 「データの場所やアクセスする端末を制限しない」（NFR-021）について, 本決定では端末をまたぐ自動同期は提供しないが, 個別端末上で動作する条件は満たす. 端末間移行が必要になれば別途 ADR を起こす.

## 廃止理由（2026-06-05 追記）

本 ADR が前提とした「単一ユーザー・端末ローカル・サーバ運用ゼロ」「Web も Android もブラウザの永続化 API（IndexedDB）に閉じる」は誤りであった. 修正後の前提は次のとおり.

- Web クライアントは自前サーバに接続し, **Web のデータは自前サーバが保持** する.
- Android クライアントは **ローカル保持 (端末内 DB) または 自前サーバ接続** を切り替え可能とする.

このため永続化の検討は次の 2 つに分けて行う.

- サーバ側永続化（Web 利用時 / Android サーバモード利用時の本体データの保存先）→ [ADR-0007](0007-server-tech-stack.md) で扱う. PostgreSQL / SQLite （サーバホスト上）/ 他の RDBMS / 軽量 KVS などをトレードオフ比較する.
- Android のローカルモード時の端末内永続化 → [ADR-0009](0009-android-client-tech-stack.md) で扱う. ネイティブの場合は Room/SQLite, Capacitor / Flutter の場合は各プラットフォームの SQLite / セキュアストレージなどが候補となる.
