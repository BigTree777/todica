## ADR-0007: サーバ側技術スタック（言語・フレームワーク・永続化・ホスティング）

> ADR-0006 で確定した「自前サーバを伴うクライアント・サーバ型」構成を前提に, サーバ側の技術スタックを選定する. 個人 1 人運用・個人負担コスト・単一ユーザー（=マルチテナント不要）を満たすこと.

- 状態: 承認（2026-06-07）
- 日付: 2026-06-05
- 決定者: プロジェクトオーナー（個人開発）

## 背景 / 状況

- 構成前提は ADR-0006 のとおり. サーバは **本人運用の単一インスタンス**. マルチテナント化しない.
- 個人 1 人開発・運用（CONSTRAINT-001）. 学習コスト・保守コスト・運用コストを最小化したい.
- データ規模は個人 1 人ぶん（数 MB 〜 数十 MB 程度を想定）. 大規模インフラ不要（project.md §12）.
- 個人負担コスト（NFR-032, CONSTRAINT-002）で運用できること. 高額サービスを前提にしない.
- データ整合性要件: リセット処理が冪等であること（NFR-020, FR-043, FR-051）. トランザクション的書き込みが扱えることが望ましい.
- Web クライアントが必ず接続する. Android のサーバモードも同じサーバに接続する.

## 検討した選択肢

### 言語・フレームワーク

| 選択肢 | 利点 | 欠点 |
| --- | --- | --- |
| A: Node.js + TypeScript（Express / Fastify / Hono など） | クライアント側と言語が揃いやすく, 型定義 / DTO を共有しやすい. エコシステムが厚い. デプロイ先の選択肢が多い（VPS / コンテナ / 無料枠 PaaS）. | ランタイム常駐型なのでメモリ常時消費. シングルプロセスのまま個人運用するとプロセス監視が必要. |
| B: Go + 標準 net/http（または chi / echo） | バイナリ 1 つで配布できデプロイが極めて軽い. メモリ消費が小さく VPS / 自宅サーバの常駐に向く. 並行性・型安全性が高い. | TypeScript の DTO 型をそのまま共有はできない（生成スキームが必要）. 個人開発者にとっての習得コストは選定者次第. |
| C: Python + FastAPI | 型ヒントと OpenAPI 自動生成が強力. データ加工が容易. | ランタイム常駐コスト + GIL によるスループット制約. 個人本人 1 人なら問題にならないが, Node.js / Go と比べて運用面のメリットは限定的. |
| D: Rust + Axum | バイナリ配布・メモリ効率・型安全性が最高水準. | 個人 1 人開発で学習・実装コストが大きい. 機能が小規模な CRUD であることを考えるとオーバースペック気味. |

### 永続化

| 選択肢 | 利点 | 欠点 |
| --- | --- | --- |
| P1: SQLite（サーバホスト上のファイル. 例: WAL モード） | サーバプロセス内蔵で運用ゼロ（別途 DB サーバ不要）. バックアップが 1 ファイルのコピーで済む. データ規模に対して十分. トランザクションあり. | 同時書き込みの並行性は低いが, 単一ユーザーなので問題にならない. 大規模スケール時は不向き（本件では不要）. |
| P2: PostgreSQL | スキーマ・制約・型・拡張性が強力. SQL の表現力が高い. | 別プロセス管理が必要. メモリを常時消費するため最安 VPS では負担が出る. 個人運用のオペレーション負荷が SQLite より高い. |
| P3: ファイルベース KVS（例: JSON ファイルを atomic write） | 実装が極めて単純. | 整合性を自力で実装する必要がある. リセット処理の冪等性（NFR-020）を担保するコードが増える. |

### ホスティング

| 選択肢 | 利点 | 欠点 |
| --- | --- | --- |
| H1: 安価な VPS（例: $5/月程度. ConoHa / Vultr / Hetzner 等） | 常駐サーバを完全に自分で握れる. SQLite ファイルの永続性が単純. SSH で運用できる. | 月額固定費が発生. OS / セキュリティ更新を自分で見る必要あり. |
| H2: 自宅サーバ / Raspberry Pi 等 | 月額のクラウド固定費ゼロ. 完全に自分の手元. | 自宅ネットワークの公開設定（DDNS / リバースプロキシ / TLS）が必要. 停電・回線断のリスク. |
| H3: 無料枠 PaaS（Fly.io / Render / Railway 等の無料枠） | 初期コストゼロ. デプロイが自動. | 無料枠は条件変動が大きく長期運用のリスクあり. アイドルスリープによる起動遅延の可能性. SQLite の永続化に追加のボリューム設定が必要なことが多い. |
| H4: クラウド BaaS（Supabase / Firebase 等の無料枠） | 認証・DB・API 自動生成までセットで来る. | 「自前サーバを本人が運用する」という ADR-0006 の方針と微妙に外れる（運用は BaaS ベンダー）. ベンダーロックインのリスク. 単一ユーザーには機能過多. |

## 決定

**次を採用する.**

- 言語・フレームワーク: **A（Node.js + TypeScript）, フレームワークは Hono に確定**
  - クライアント側と言語が揃うことで, タスク・プロジェクト等の DTO 型をリポジトリ内で共有できる（個人 1 人開発の生産性に効く）.
  - エコシステムが厚く, OpenAPI 自動生成・テストランナー・Lint まで揃う.
  - デプロイ先の選択肢が広い.
- 永続化: **P1（SQLite, サーバプロセス内蔵 / WAL モード）**
  - データ規模に対して十分. 別プロセス管理不要で個人運用に最適.
  - バックアップが 1 ファイルのコピーで済むため, NFR-032（個人負担コスト）と整合.
  - トランザクションが使えるためリセット処理の冪等性（NFR-020）を構造的に満たしやすい.
- ホスティング: **H1（安価な VPS, $5/月程度）を第一候補, H2（自宅サーバ）を予備候補**
  - 長期運用の安定性と公開しやすさのバランスが最も良い.
  - H2 は固定費ゼロだが運用の手間と回線可用性のトレードオフ. 個人本人の運用スタイル次第で切替可.
  - H3 / H4 は短期検証や費用ゼロ要求が強い場合の代替.

代替案の採用条件:

- 「個人開発者として Go 経験が長く, バイナリ 1 つ運用が好み」なら B（Go + chi/echo）+ P1（SQLite）+ H1 / H2 が有力.
- 「PostgreSQL 経験が長く SQL 制約を厚く効かせたい」なら P2（PostgreSQL）に切替. ホスティングは H1（VPS）または H4（Supabase）になりやすい.
- 「初期コストゼロを最優先したい」なら H3（無料枠 PaaS）または H2（自宅）.

## 結果 / 影響

- 良い影響
  - 個人 1 人で運用できる最小構成 + 個人負担コストの範囲に収まる.
  - トランザクション境界をサーバ側に集約でき, 「リセット処理の冪等性」が API レイヤ単位で保証できる.
  - データ規模が小さいため, 単一プロセス + SQLite で十分にスケールする.
- トレードオフ / 注意点
  - 推奨案では PostgreSQL は採用しない. ただし将来的に複雑なクエリ要件が出れば再評価する（ADR を改めて起こす）.
  - フレームワークは Hono に確定（軽量・現代的・Web 標準準拠・TypeScript ファースト設計のため）.
  - 認証方式・API プロトコルの詳細は [ADR-0010](0010-api-design.md) で扱う.
