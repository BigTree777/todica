## ADR-0010: API 設計の方針（プロトコル・認証・バージョニング）

> ADR-0006 で確定した「自前サーバ + クライアントの API 通信」を前提に, 通信プロトコル・認証・バージョニングの方針を決める. ADR-0005（外部 API なし）の Supersede.

- 状態: 承認（2026-06-07）
- 日付: 2026-06-05
- 決定者: プロジェクトオーナー（個人開発）

## 背景 / 状況

- 配布構成は ADR-0006 のとおり.
  - Web クライアントは必ずサーバに接続.
  - Android クライアントはサーバモード時のみ接続.
  - サーバは本人運用の単一インスタンス. マルチテナント化しない（CORE-2, project.md §8 Out of Scope "共有 SaaS としてのホスティング提供"）.
- 「外部 API」ではなく「本人運用サーバとそのクライアントの間の内部 API」として位置づける.
- 単一ユーザーのため認証は最小限で良い. ただし HTTPS 必須 + 何らかのアクセス制御は必要（インターネット越しに公開する場合, 第三者の書き込みを防ぐため）.

## 検討した選択肢

### 通信プロトコル

| 選択肢 | 利点 | 欠点 |
| --- | --- | --- |
| Pr1: REST (HTTP + JSON) | 最も枯れていて事例豊富. OpenAPI 定義・コード生成・ブラウザのキャッシュ機構と相性が良い. 個人 1 人運用の保守コストが最小. | リソース指向の設計に縛られ, 一部の操作（例: リセット処理を 1 アクションで起動）が表現上やや不格好. |
| Pr2: GraphQL | クエリ柔軟. クライアントが必要なフィールドだけ取れる. | 個人 1 人運用にはオーバーキル. キャッシュ層が REST より複雑. データ規模が小さいので恩恵が薄い. |
| Pr3: tRPC / 単純 RPC over JSON | サーバ・クライアントが TypeScript の場合, 関数呼び出しのように型安全に呼べる. 個人開発の生産性が高い. | サーバ・クライアントが同一言語前提. Android がネイティブ実装になる場合（ADR-0009 案 B）はそのまま使えない. |

### 認証

| 選択肢 | 利点 | 欠点 |
| --- | --- | --- |
| Au1: 単一の Bearer トークン（事前共有秘密. クライアントに 1 つだけ持たせる） | 最小実装. 単一ユーザー前提に合う. ローテーションも単純. | トークン漏洩時の影響範囲はサーバ全体. ただし単一ユーザーなのでアカウントごとの分離は元々ない. |
| Au2: 単純なパスワード認証 + セッション Cookie / JWT | 「クライアント側でパスワードを入力 → セッション発行」の形が直感的. 通信内容に通常のセッション管理が乗る. | セッション管理の実装コストが Au1 より高い. 個人運用で意味があるかは判断次第. |
| Au3: 認証なし（プライベートネットワークのみで公開. VPN / 自宅 LAN 等） | 認証ロジックなし. 最も単純. | サーバを公開ネットワークに置けない. Web を外出先から使う想定なら不可. |

### バージョニング

| 選択肢 | 利点 | 欠点 |
| --- | --- | --- |
| V1: URL パス / `Accept` ヘッダで明示的にバージョンを切る（例: `/api/v1/...`） | 破壊的変更時にクライアントを段階的に移行できる. | バージョンを切るたびに保守対象が増える. |
| V2: バージョンを切らず, 常に最新 1 つだけ提供する. 破壊的変更時はクライアント / サーバを同時更新する. | 個人本人 1 人運用なら現実的. 保守対象が常に 1 つ. | クライアント・サーバを同時更新する規律が必要. |

## 決定

**次を採用する.**

- プロトコル: **Pr1（REST + JSON）**.
- 認証: **Au1（単一の Bearer トークン）** を採用. Au3（認証なし + プライベート公開）は「自宅 LAN / VPN 越しでしか使わない」運用時のみのオプションとして残す.
- バージョニング: **V1（明示的なパスバージョン. 初期は `/api/v1/...`）**.

## 冪等性キーと楽観ロック（[ADR-0008](0008-web-client-tech-stack.md) 由来）

PWA + オフライン書込キュー（[ADR-0008](0008-web-client-tech-stack.md)）のため, 書き込み API は以下のヘッダを扱う.

### 冪等性キー: `Idempotency-Key` ヘッダ

- **対象**: 書き込み系 API すべて（POST / PATCH / PUT / DELETE）.
- **値**: 操作対象エンティティの ID（UUID v4）. クライアントが先行採番する.
- **サーバ側の処理**: 直近一定期間（24 時間程度を目安. 実装段階で確定）に同じ `Idempotency-Key` で同等の要求を処理済みなら, 新規実行せず保存しておいた応答を返す.
- **目的**: オフライン書込キューが接続復帰時に再送した場合の二重登録を防ぐ.

### 楽観ロック: `If-Match` ヘッダ

- **対象**: 既存エンティティへの更新系 API（PATCH / PUT / POST `:id/complete` 等の状態遷移）.
- **値**: クライアントが保持するエンティティの `version` 整数値.
- **サーバ側の処理**:
  - 一致: 通常実行し, レスポンスで新しい `version` を返す（`version` を +1 する）.
  - 不一致: **412 Precondition Failed** を返し, 現行のエンティティ全体をレスポンスボディに含める.
- **クライアント側の処理**: 412 を受けたら, UI で「サーバ値で上書き / クライアント値で再送（強制）」をユーザーに確認する.

## リソース概観（詳細は feature spec）

本 ADR では HTTP メソッドとリソースの輪郭だけ示す. 詳細パス・パラメータは feature spec で確定.

| リソース | 主な操作 | 由来 FR |
| --- | --- | --- |
| `/api/v1/tasks` | 一覧取得 / 起票（POST） / 編集（PATCH） / 完了（POST `:id/complete`） / 削除（DELETE `:id`. ゴミ箱へ移す semantics） / ゴミ箱からの復元（POST `:id/restore`） | FR-001 〜 FR-009, FR-014, FR-061 |
| `/api/v1/today` | 今日ビュー（タスク一覧 + 並び順 + フォーカス参照）取得 | FR-010 〜 FR-013 |
| `/api/v1/focus` | 現在のタスク参照（GET / PUT） | FR-012, FR-013 |
| `/api/v1/projects` | 一覧 / 作成 / 編集 / 削除 / 復元 | FR-020, FR-022, FR-060, FR-061 |
| `/api/v1/routines` | 一覧 / 作成 / 編集 / 削除 / 復元 | FR-030, FR-031, FR-035 |
| `/api/v1/trash` | 一覧 / 手動空にする | FR-061, FR-062 |
| `/api/v1/counter` | 今日の完了数取得 | FR-040 |
| `/api/v1/settings` | 境界時刻設定の取得 / 更新 | FR-042 |
| `/api/v1/reset` | リセット処理の手動起動（保守用. 通常は境界時刻で自動起動）| FR-043, NFR-020 |

- 「今日の完了数」「フォーカス」「設定」は単一レコード（architecture/database/schema.md と整合）.
- API レスポンスは 1 ユーザー前提のスキーマ. ユーザー ID / テナント ID 等のフィールドは持たない（CORE-2 と整合）.

## バージョニング・互換性運用

- 破壊的変更を入れる場合は新しいパスバージョン（`/api/v2/...`）を導入し, 旧バージョンを一定期間並行で維持する（個人運用なので「一定期間」は短くて構わない. 目安: 1 週間〜1 か月）.
- 非破壊的なフィールド追加は同一バージョンで行う.
- スキーマ変更とサーバ DB マイグレーションは architecture/database/migration-policy.md と連動する.

## OpenAPI 定義の扱い

- リポジトリに `docs/developer/architecture/api/openapi.yaml` を **復元する**. 構成のスケルトンだけを置き, 詳細は feature spec ごとに追記していく.
- 本 ADR では「OpenAPI を持つ」という方針までを確定し, 具体的なスキーマは feature spec / implementer の責務とする.

## 結果 / 影響

- 良い影響
  - 単一ユーザー前提を維持したまま, 通信レイヤだけクライアント・サーバ型に再整理できる.
  - Web ↔ Android の両方が同じ API を叩く構造になり, サーバ側のロジックを 1 か所に集約できる.
- トレードオフ / 注意点
  - 認証は最小実装で済ませる方針のため, Bearer トークンの漏洩管理（リポジトリへの誤コミット防止・ローテーション手段）はセキュリティ運用の責任になる.
  - 外部公開 API ではない（共有 SaaS にしない）ことを README / architecture/api/overview.md で繰り返し明示する必要がある.
