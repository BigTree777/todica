# Todica 要件定義書

> 本書は, [`project.md`](project.md)（プロジェクト憲章）から派生するプロジェクト横断の要件定義書である. 
> 各機能の詳細仕様（`features/<feature-name>/spec.md`）は本書を上位の根拠として参照する. 
> 本書を変更する場合は project.md との整合を確認すること. 

- バージョン: 1.0.0
- 上位文書: [`project.md`](project.md)
- 派生先: `features/<feature-name>/spec.md`（順次切り出し）

## 1. 本書の位置づけと読み方

- 本書はプロジェクト全体に通底する**プロダクト要件**を, FR（機能要件）・NFR（非機能要件）・スコープ外として整理する. 
- ID 体系
  - 機能要件: `FR-XXX`（3 桁連番）
  - 非機能要件: `NFR-XXX`（3 桁連番）
  - スコープ外: `OOS-XXX`（3 桁連番）
  - 未決事項: `UNRESOLVED-XXX`（3 桁連番）
- 各要件には, 由来する project.md の該当箇所（節番号 / 識別子）を inline で示す（トレーサビリティ）. 
- 受け入れ基準（Given/When/Then）は本書には書かない. 各 feature の `spec.md` で詳細化する. 本書は "何を満たすべきか" のレベルにとどめる. 
- 本書と project.md が矛盾した場合は project.md を優先する. 

## 2. WHAT / HOW 切り分け方針（本書の境界）

本書は "ユーザーが知覚する WHAT" のみを扱う. "システム側の HOW"（同じ WHAT を満たすための実現手段の選択）は基本設計以降, すなわち各 feature の `spec.md` 以降に委ねる. 

### 基準

> **ユーザーが触ったときに感じ取れる挙動・制約・性質か? それとも同じユーザー体験を満たす中で「中の人」が選ぶ実現方法か?**

- 触って感じ取れる → **要件（本書で確定）**. 
- 同じユーザー体験を異なる手段で実現できる → **手段選択（本書では扱わない. feature spec / architecture / plan で確定）**. 

### 本書に書く / 書かないの具体例

- 書く: 「今日ビューには "次に取り組む 1 つ" を一意に定める並びが存在する」（capability 側）, および「同じデータ状態なら常に同じ順序になる（予測可能性）」（品質側）. 
- 書かない: 「同一優先度は作成日時昇順とする」「ID の二次キーで安定ソートする」などの具体ルール. 
- 書く: 「『今日』の境界は時刻で切り替わり, ユーザーは境界時刻を設定変更できる」. 
- 書かない: 「既定値は 04:00 とする」などの具体値. 
- 書く: 「タスクの優先度・期限・名称は変更できる / できない」という最終的な可否. 
- 書かない: 「編集 UI は長押しで開く」「内部でイベントソーシングを使う」などの実現方法. 

### 規律

- 要件は "結果（何が成り立つか）" を書き, "手段（どう成り立たせるか）" を書かない. 
- 同じ結果を別手段で達成できる場合, 手段選択は feature spec 側の責務とし, 本書はそれを縛らない. 
- 仕様策定の過程で本書と矛盾する手段が必要になった場合は, 本書 → project.md の順に整合を再評価する. 

## 3. 用語

- **プロジェクト**: タスクに文脈を与える集合. 役割は "タスクに文脈を与える" こと（参考: project.md §8 In Scope）. 
- **タスク**: 最小単位の作業項目（参考: project.md §8 In Scope, "プロジェクト → タスクの 2 階層"）. 
- **プロジェクト外タスク（単独タスク）**: いずれのプロジェクトにも属さないタスク（参考: project.md §8 In Scope）. 
- **ルーティン**: 指定曜日に自動でタスク化される独立概念（参考: project.md §8 In Scope, UC-004）. 
- **ルーティン由来タスク**: ルーティン定義から自動生成された当日のタスク. 
- **今日ビュー**: 期限=今日のタスクを優先度順に表示する画面. アプリの入口（参考: project.md §8 In Scope, UC-001）. 
- **現在のタスク（フォーカス）**: 今日のタスクから 1 つを選び大きく表示している状態（参考: project.md §8 In Scope, UC-001）. 
- **ゴミ箱**: 削除・完了を経由する一時保管場所（参考: project.md §8 In Scope）. 復元の可否は FR-061, 清算ルールは FR-062 に従う. 
- **リセット**: ユーザー設定の境界時刻を境に "今日" を切り替え, カウントとゴミ箱を清算する処理（参考: project.md §8 In Scope）. 
- **境界時刻**: "今日" と "翌日" を切り替える時刻. ユーザーが設定する（参考: project.md §8 In Scope "リセット時刻はユーザー設定可"）. 

## 4. ステークホルダーと利用環境前提

- 利用者: 単一ユーザー本人のみ（参考: project.md §3 CORE-2, §8 In Scope "単一ユーザーでの利用"）. 
- 開発者・運用者: 個人 1 人（参考: project.md §11 CONSTRAINT-001）. 
- 配布形態: Web アプリおよび Android アプリの 2 形態を取る. 
  - Web アプリ: 個人利用および OSS 公開における初期 / 並行プラットフォーム. 同一コードベースから先行して提供できる形態として位置づける（参考: project.md §12 Technical Feasibility "Web から Android への展開とストア公開は確立した手法で実現できる", §1 Project Summary "（オープンソース）"）. 
  - Android アプリ: プロジェクトゴールとしての最終的な配布形態. 配布先は Google Play Store（参考: project.md §1 Project Summary, §11 CONSTRAINT-003, §12 Technical Feasibility, §13 RISK-002）. 

## 5. 機能要件（FR）

> 各 FR は project.md の Use Cases（UC-001 〜 UC-005）および §8 In Scope を根拠とする. 

### 5.1 タスク管理（起票・編集・削除・完了）

- **FR-001: タスクを起票できる**
  - 利用者は思いついた作業を, 粒度を問わずに記録できること. 
  - 起票時, プロジェクトに紐づけても, 紐づけずに単独タスクとしてもよい. 
  - 関連: NFR-010. 
  - 由来: project.md §7 UC-002, §6 Value Proposition "気軽・高速にタスク化できる", §8 In Scope "プロジェクト外タスク". 
- **FR-002: タスクの期限の値域は今日 / 明日のみ**
  - 起票時の既定値は "今日". 設定可能な値は "今日" または "明日" の 2 値に限定する. 
  - 登録済みタスクに対する変更操作は FR-005 で規定する（本 FR は値域の規定）. 
  - 関連: FR-005. 
  - 由来: project.md §6 Value Proposition "設定で迷わない", §7 UC-002, §8 In Scope "タスク管理（期限つき, 既定=今日, 設定は今日 / 明日のみ）", §8 Out of Scope "3 日以降の期限指定". 
- **FR-003: タスクに 3 段階の優先度を設定できる**
  - 値は "最優先 / 普通 / 後回し" の 3 段階. それ以外の優先度は持たない. 
  - 由来: project.md §8 In Scope "タスクの優先度（3 段階）". 
- **FR-004: タスクの優先度は登録後に変更できる**
  - 既に登録済みのタスクについても, 利用者は任意に優先度を 3 段階の中で変更できる. 
  - 由来: project.md §7 UC-002 "後から優先度を変えたり, 不要なら削除したりできる". 
- **FR-005: タスクの期限は今日 ↔ 明日で変更できる**
  - 登録済みタスクの期限を, 今日 / 明日の 2 値の範囲内で切り替えられる.（値域は FR-002 に準ずる）. 
  - 関連: FR-002, FR-014（今日ビューからの今日 → 明日 切替）. 
  - 由来: project.md §7 UC-001 "今日できないものは期限を明日にずらす", §7 UC-002, §8 In Scope "設定は今日 / 明日のみ". 
- **FR-006: タスクを完了できる（完了アクション）**
  - 完了されたタスクはゴミ箱へ移動する. 
  - 完了は "今日の完了タスク数" のカウントに +1 として反映される. 
  - 由来: project.md §8 In Scope "完了アクション（ゴミ箱へ移動＋今日の完了数 +1）", §7 UC-001. 
- **FR-007: タスクを削除できる（削除アクション）**
  - 削除されたタスクはゴミ箱へ移動する. 
  - 削除は "今日の完了タスク数" のカウントには含めない. 
  - 由来: project.md §8 In Scope "削除アクション（ゴミ箱へ移動, カウントには含めない）". 
- **FR-008: タスクは "プロジェクト → タスク" の 2 階層に固定される**
  - サブタスク・ネスト構造を持たない. タスクが最小単位である. 
  - 由来: project.md §8 In Scope "プロジェクト → タスクの 2 階層（タスクが最小単位）", §8 Out of Scope "サブタスク / サブプロジェクト / ネスト". 
- **FR-009: タスクの名称は登録後に変更できる**
  - 既に登録済みのタスクについても, 利用者は任意に名称を変更できる. 
  - 軽微な修正（誤字脱字・表現の微調整など）のために削除→再登録を強いる UX を避ける目的. 
  - 関連: FR-004（優先度の変更）, FR-005（期限の変更）. 
  - 由来: ユーザー決定（旧 UNRESOLVED-001）, project.md §7 UC-002 "後から優先度を変えたり, 不要なら削除したりできる" の延長として整合. 

### 5.2 今日ビューと現在のタスク（フォーカス）

- **FR-010: 今日ビューを入口として表示する**
  - アプリを開いたときの入口は "今日" ビューとする. 
  - 全プロジェクト横断 + プロジェクト外 + ルーティン由来を対象に, 期限=今日のタスクを優先度順に表示する. 
  - 由来: project.md §7 UC-001, §8 In Scope "今日ビュー = アプリの入口". 
- **FR-011: 今日ビューには全体順序（"次の 1 つ" を一意に定める並び）が存在する**
  - 並び順は優先度を主キーとし, 利用者が並べ替え操作を行わなくても "次の 1 つ" が一意に定まる構造を持つ. 
  - 具体的な順序規則（同一優先度内の二次キーなど）は本書で規定せず, feature spec で確定する. 
  - 並び順の一貫性・予測可能性（同じデータ状態なら常に同じ順序になる, という品質側の保証）は NFR-013 に委ねる. 
  - 関連: NFR-013, NFR-001（単一ワークフロー）. 
  - 由来: project.md §6 Value Proposition "今やる1つに集中できる", §7 UC-001. 
- **FR-012: 今日のタスクから 1 つを "現在のタスク" に選べる**
  - 選択された 1 つはフォーカスビューで大きく表示される. 
  - 同時に "現在のタスク" になれるのは 1 つだけとする. 
  - 由来: project.md §6 Value Proposition "今やる1つに集中できる", §7 UC-001. 
- **FR-013: 現在のタスクが完了されたら自動で次へ繰り上げる**
  - 完了アクション後, 残った今日のタスクのうち FR-011 の並び順における最上位を新たな "現在のタスク" にする. 
  - 由来: project.md §7 UC-001 "終わったら完了アクションで消化する. 自動で次（優先度最上位）が現在のタスクになる". 
- **FR-014: 今日のタスクは期限を明日にずらせる**
  - FR-005 の操作のうち, 今日 → 明日 への切替は今日ビューから直接行える. 
  - 関連: FR-005. 
  - 由来: project.md §7 UC-001 "今日できないものは期限を明日にずらす". 

### 5.3 プロジェクト管理

- **FR-020: プロジェクトを作成・削除できる**
  - 複数プロジェクトを並行して持てる. 
  - 削除は FR-060 に従う（ゴミ箱経由）. 
  - 関連: FR-060. 
  - 由来: project.md §7 UC-003, §8 In Scope "プロジェクトの管理". 
- **FR-021: プロジェクトのアーカイブ機能は持たない**
  - 区切りがついたプロジェクトは削除する. "残して整理する" 概念を持たない. 
  - 由来: project.md §8 Out of Scope "プロジェクトのアーカイブ". 
- **FR-022: プロジェクトの名称は作成後に変更できる**
  - 既に作成済みのプロジェクトについても, 利用者は任意に名称を変更できる. 
  - 軽微な修正のために再作成（新規作成 + 旧プロジェクト削除）を強いる UX を避ける目的. 
  - 名称変更によって既存タスクとの紐付けは保持される. 
  - 由来: ユーザー決定（旧 UNRESOLVED-002）, project.md §7 UC-003 と整合. 

### 5.4 ルーティン

- **FR-030: ルーティンを定義できる**
  - ルーティンは独立概念として定義し, 生成する曜日と既定優先度を指定できる. 
  - 由来: project.md §7 UC-004, §8 In Scope "ルーティン機能（独立概念）". 
- **FR-031: ルーティンは指定曜日に自動で今日のタスクとして生成される**
  - 生成されたタスクは今日ビューに現れる. 
  - 由来: project.md §7 UC-004, §8 In Scope. 
- **FR-033: ルーティン由来タスクは翌日に持ち越さない**
  - 当日中に実施されなかったルーティン由来タスクは翌日には存在しない.
  - 通常タスクの "今日 → 明日" 切替（FR-005 / FR-014）はルーティン由来タスクには適用されない（手動経路の持ち越しを認めない）.
  - 通常タスクの自動繰越（FR-051）の対象外である（自動経路の持ち越しを認めない）.
  - 関連: FR-005, FR-014, FR-043（リセット処理）, FR-051.
  - 由来: project.md §7 UC-004 "翌日に持ち越さない" "実施しなくても日付の切替で自動削除される", §8 In Scope "未完了タスクの自動繰越（ルーティン由来を除く）".
- **FR-034: ルーティンの実施履歴・ストリークは保持しない**
  - 由来: project.md §8 Out of Scope "ルーティンの実施履歴 / ストリーク", §7 UC-004 "履歴なし". 
- **FR-035: ルーティンの定義（名称・生成曜日・既定優先度）は定義後に変更できる**
  - 既に定義済みのルーティンについても, 利用者は任意に名称・生成曜日・既定優先度を変更できる. 
  - 設定の部分変更ニーズに対応する目的. project.md §7 UC-004 の「ルーティンを一度だけ定義する」の語感とは厳密には外れるが, 運用上の修正コストを下げる判断としてユーザー承認済み. 
  - 関連: FR-031（指定曜日に自動生成）, FR-033（翌日に持ち越さない）. 
  - 由来: ユーザー決定（旧 UNRESOLVED-003）, project.md §7 UC-004 と整合（運用上の修正コストを優先する判断としてユーザー承認）. 

### 5.5 今日の完了数カウントとリセット

- **FR-040: "今日の完了タスク数" を表示する**
  - 当日の完了タスク数を利用者に提示する. 
  - カウントの増減ルールは FR-006 / FR-007 で規定する. 
  - 関連: FR-006, FR-007. 
  - 由来: project.md §8 In Scope "今日の完了タスク数 のカウント表示". 
- **FR-041: "今日" の境界は時刻で切り替わる**
  - "今日" と "翌日" の切替は, 定められた境界時刻をもって行う. 
  - 関連: FR-062（境界時刻に作動するゴミ箱の自動完全削除処理）. 
  - 由来: project.md §8 In Scope "日次リセット". 
- **FR-042: 境界時刻はユーザーが設定変更できる**
  - 利用者は境界時刻を任意の値に設定できる. 
  - 境界時刻の具体的な既定値や設定可能粒度は本書では規定せず, feature spec で確定する. 
  - 関連: FR-062（境界時刻に作動するゴミ箱の自動完全削除処理）. 
  - 由来: project.md §8 In Scope "リセット時刻はユーザー設定可". 
- **FR-043: 境界時刻におけるリセット処理で "今日" を切り替える**
  - 境界時刻を境に, "今日の完了タスク数" の 0 クリアと未完了タスクの繰越処理（FR-051）を起動する. 
  - ゴミ箱の自動完全削除は FR-062, ルーティン由来タスクの未繰越は FR-033 を参照. 
  - 関連: FR-062, FR-033, FR-051. 
  - 由来: project.md §8 In Scope "日次リセット". 

### 5.6 自動繰越

- **FR-051: 未完了タスクを日跨ぎで自動繰越する**
  - 期限=今日のまま完了されなかった通常タスクは, リセット時に期限=今日・優先度=最優先で次の日に持ち越す. 
  - ルーティン由来タスクは対象外（FR-033）. 
  - 由来: project.md §8 In Scope "未完了タスクの自動繰越（日跨ぎで期限=今日・優先度=最優先. ルーティン由来を除く）". 

### 5.7 ゴミ箱

- **FR-060: すべての削除・完了はゴミ箱を経由する**
  - 由来: project.md §8 In Scope "ゴミ箱（削除はすべてゴミ箱経由）", FR-006 / FR-007. 
- **FR-061: ゴミ箱から復元できる**
  - ゴミ箱内の対象を元の状態（タスク・プロジェクト）に戻せる. 
  - ゴミ箱が空になった後は復元できない（復元対象が存在しないため）. 
  - 関連: FR-062（リセット時の自動完全削除）. "同日内に限る" という時間的限定は本 FR では規定しない（FR-062 によりリセットを境にゴミ箱が空になる結果として導かれる派生事実であるため）. 
  - 由来: project.md §8 In Scope "同日内は復元可", §7 UC-003. なお project.md 原典の "同日内" という時間的限定は, 本書では FR-062（境界時刻でのゴミ箱完全削除）の帰結として扱う. 
- **FR-062: リセット時にゴミ箱を自動完全削除する**
  - 加えて, ユーザーが手動で "空にする" を実行できる. 
  - 由来: project.md §8 In Scope "リセット時に自動完全削除＋手動 空にする". 

### 5.8 PM ツールとの併用

- **FR-070: PM ツール（Asana 等）との自動連携は持たない**
  - 利用者が手で書き写す運用を前提とし, 双方は独立して動作する. 
  - 由来: project.md §7 UC-005 "両者は独立しており, 自動連携はしない", §8 Out of Scope "外部連携 / 自動化ルール". 

## 6. 非機能要件（NFR）

### 6.1 設計原則由来（CORE-1 / CORE-2）

- **NFR-001: 単一ワークフローを強制する設計であること**
  - "プロジェクト作成 → タスクを最小手数で足す → 1 つ選んで集中 → 完了で次へ" の一本道を, UI と機能の双方で強制する. 
  - カスタマイズ設定画面・ビュー切替・ユーザー定義ソート等の "選択肢を増やす機能" を持たない（スコープ外と整合）. 
  - 由来: project.md §3 CORE-1, §3 Design North Star 1・3, §8 Out of Scope. 
- **NFR-002: マルチユーザー概念を一切持たないこと**
  - 担当者・権限・共有・コラボに関する概念をデータモデルおよび UI から排除する. 
  - 由来: project.md §3 CORE-2, §8 In Scope "単一ユーザーでの利用", §8 Out of Scope "複数ユーザー / チーム機能". 

### 6.2 ユーザビリティ・操作性

- **NFR-010: タスク起票は最小手数で完了できること**
  - 既定期限=今日, 既定プロジェクト未指定（単独タスクとして登録可）の前提で, 名前入力以外の必須項目を増やさない. 
  - 関連: FR-001. 
  - 由来: project.md §2 Purpose "タスク化の手間を限界まで減らす", §6 Value Proposition "気軽・高速にタスク化できる", §7 UC-002. 
- **NFR-011: 現在のタスクは "大きく" 単独で表示されること**
  - フォーカス時, 他のタスクが視認の主役にならない表示にする. 
  - 由来: project.md §6 Value Proposition "それだけが大きく表示される", §7 UC-001. 
- **NFR-012: 設定項目を最小限に抑えること**
  - 設定可能な項目は FR-002（期限の値域）・FR-003（優先度の段階）・FR-042（境界時刻）に列挙されたものに限定し, それ以外のカスタマイズ設定画面を設けない. 
  - 由来: project.md §6 Value Proposition "設定で迷わない", §8 Out of Scope "カスタマイズ設定画面". 
- **NFR-013: 今日ビューの並び順は予測可能で一貫していること**
  - 同じデータ状態から表示すれば, 常に同じ順序になる（決定論性）. 
  - 利用者は順序の揺らぎに惑わされず, "次の 1 つ" を信頼できる. 
  - 関連: FR-011（並びの構造そのものの規定）. 
  - 由来: project.md §6 Value Proposition "今やる1つに集中できる", §7 UC-001, NFR-001（単一ワークフロー）. 

### 6.3 データ・整合性

- **NFR-020: タスク・プロジェクト・ルーティン・ゴミ箱・完了数カウントは, リセット処理を境とした日次の状態として整合的に管理されること**
  - リセット処理が中断・再起動を挟んでも, 二重繰越・二重リセットを起こさないこと. 
  - 由来: project.md §8 In Scope "日次リセット", "未完了タスクの自動繰越", FR-043 / FR-051. 
- **NFR-021: 単一ユーザーのデータとして成立すること**
  - 共有 SaaS としてのホスティングや, 端末間同期を前提としない. 
  - データの場所やアクセスする端末を制限しない.
  - 由来: project.md §8 Out of Scope "共有 SaaS としてのホスティング提供", §3 CORE-2. 

### 6.4 配布・運用環境

- **NFR-030: Google Play Store の規約・審査要件を満たせる実装・運用であること**
  - 最終目標は Google Play Store への公開である. 
  - 由来: project.md §1 Project Summary, §11 CONSTRAINT-003. 
- **NFR-031: 同一コードベースから Android アプリとして配布できる構成であること**
  - "コードを大きく作り直さずに Android アプリ化できる" を前提とし, 早期に Android ビルド・公開フローを小さく試して検証する. 
  - 由来: project.md §10 ASM-003, §13 RISK-002 の対策. 
- **NFR-032: 個人負担で運用できるコスト構造であること**
  - 高額な有料サービス・ライセンスを前提としない. 大規模インフラを必要としない. 
  - 由来: project.md §11 CONSTRAINT-002, §12 Operational Feasibility. 

### 6.5 オープンソース性

- **NFR-050: コードを公開可能な形で開発すること**
  - ライセンス・依存関係・秘密情報の取扱いを, 公開を前提として整える. 
  - 由来: project.md §1 Project Summary "（オープンソース）".
