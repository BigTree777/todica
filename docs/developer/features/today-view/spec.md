# 仕様: 今日ビュー（入口・優先度順表示・"次の 1 つ" の一意化）

- 状態: ドラフト
- 関連: [`../../planning/backlog.md`](../../planning/backlog.md) BL-005
- 由来要件: FR-010（今日ビュー = アプリの入り口）/ FR-011（全体順序の存在）
- 関連 NFR: NFR-001（単一ワークフロー強制）/ NFR-013（並び順の予測可能性）

## 背景 / 課題

Todica の中核体験は UC-001「今日のループを回す」であり, 利用者は **アプリを開いた瞬間に "今日やる残り" の全体像を上から消化できる順で見られる** ことを期待する. ところが現状は以下のような状態である.

- BL-001（task-crud）で `TodayView` コンポーネントが立ち上がり, BL-002 / BL-003 で優先度 UI と完了ボタンが追加された.
- しかしその並び順は **暫定実装**（`dueDate (today→tomorrow) → priority → createdAt`）であり, **「今日ビュー」の表示対象が today と tomorrow の両方** という, FR-010 と一致しない振る舞いになっている（BL-001 plan.md §影響範囲 §UI で明示された暫定）.
- サーバ API も `GET /api/v1/tasks` を全件取得した上でクライアント側で 3 段ソートする構造で, FR-010 の「今日ビュー = 期限=今日のタスクを優先度順で表示」という意味論を API 表現に反映できていない.
- `/today` エンドポイントは `openapi.yaml` に骨格があるが未実装. tag だけ存在する.
- 並び順は決定論的だが, 「次の 1 つ」を **明示的に位置付ける概念** がまだ表現上存在せず, BL-006（FocusSelection）が未選択の状態でも「上から一意に取り出せる」ことを利用者・後段機能の両方に保証するインターフェイスがない.

BL-005 は **FR-010 / FR-011 / NFR-013 を満たす形に今日ビューを本実装** することで, このギャップを埋める. 並び順仕様を確定させ, "次の 1 つ" が一意に定まる構造を明示し, 今日ビューが「期限=今日のタスクの場」になるよう範囲を絞る.

## ゴール / 非ゴール

### ゴール

- **アプリの入口を今日ビューに固定する**（FR-010）. Web クライアントが起動した直後（`/` パス）に表示されるのは今日ビューであり, 設定でこの挙動を変更できない（NFR-001）.
- **今日ビューの表示対象を `dueDate = "today"` のタスクのみに絞る**（FR-010）. tomorrow タスクは今日ビューには出さない.
- **並び順の本仕様を確定する**（FR-011 / NFR-013）.
  - 主キー: 優先度（highest → normal → later）.
  - 副キー（同一優先度内の決定論性確保）は本 spec で確定し, 後段のテスト設計者がそのまま検証できる粒度で書く.
  - 同じデータ状態を入力すれば常に同じ順序になる（決定論性 = NFR-013）.
- **"次の 1 つ" が一意に定まることをインターフェイスとして明示する**（FR-011）. 今日ビューが返す並びの先頭要素を, 利用者にも後段機能（BL-006 FocusSelection）にも参照可能な形で公開する.
- **`/today` エンドポイントを実装する**（API 上で「今日ビュー」を 1 級概念として表現する. 詳細方針は plan.md で確定）.
- **既存 `TodayView` コンポーネントの責務を本仕様に揃える**. 現状の `today + tomorrow` 表示 / 3 段ソートを本決定の並び順・対象に置き換える.

### 非ゴール

- **「現在のタスク」(FocusSelection) の選択・大表示** → BL-006 / FR-012 / FR-013 / NFR-011 の責務. 本 feature は「並びの先頭が一意に決まる」「次の 1 つを参照可能な形で公開する」までを担当し, **その選択状態を保持する FocusSelection エンティティの読み書きはしない**.
- **完了による次タスクへの自動繰り上げ** → BL-006 / FR-013 の責務. 本 feature の今日ビュー API / UI は, タスク完了時には単純に該当タスクが一覧から外れた結果として並びの先頭が「次のタスク」に切り替わることのみ保証する（FocusSelection を更新しない）.
- **tomorrow タスクの閲覧ビューの新設** → 本 feature は「今日ビューに tomorrow を出さない」ことのみ規定する. tomorrow タスクの確認方法は本 feature では UI を提供せず, 既存の `GET /api/v1/tasks?trashed=false` で取得できる前提とする（BL-007 の「今日 → 明日」切替で動かしたタスクの行き先表示は将来検討）.
- **完了数カウント表示** → BL-008 / FR-040 の責務.
- **境界時刻（日付の切替）に基づく "今日" 判定の本実装** → BL-009 / BL-010 / FR-041〜043. 本 feature は「`dueDate = "today"` を「今日」とみなす」というデータモデル側の意味論のみに依拠する. 境界時刻による値の遷移処理（自動繰越・カウントリセット）は別 BL の責務.
- **今日 → 明日 の期限切替導線**（FR-014） → BL-007 の責務. ただし BL-001 で実装済みの「今日ビューから dueDate を切り替えるトグル」自体は本 feature の対象範囲（今日ビュー）に存在するため, 仕様上の整合だけ確認する（切替後にそのタスクが今日ビューから消えること）.
- **ルーティン由来タスクの今日ビューへの自動投入** → BL-017 / FR-031. 本 feature は `origin` で挙動を分岐させず, `dueDate = "today"` のタスクであれば routine 由来も manual も同じ並びに乗せる, ことのみ規定する.
- **PWA・オフライン化** → BL-018.

## 要件

### 機能要件

本機能で実装する FR は以下のとおり.

- **FR-010（今日ビュー = アプリの入り口）**
  - Web クライアント起動時, ユーザーが最初に目にする画面は今日ビューであり, ルーティング上の `/` パスが今日ビューに紐づく.
  - 今日ビューに表示するタスクは **`dueDate = "today"` かつ `trashedAt = null` のもののみ**.
  - 全プロジェクト横断（projectId = null も含む）であり, ルーティン由来タスク（origin = "routine"）も区別なく含める（既存仕様の維持）.
- **FR-011（"次の 1 つ" を一意に定める並びの存在）**
  - 今日ビューは決定論的な全体順序を返し, 先頭要素が「次の 1 つ」として一意に定まる.
  - 並び順の規則:
    1. **第一キー: 優先度** `highest`（最優先） → `normal`（普通） → `later`（後回し）の昇順.
    2. **第二キー: `createdAt`** の昇順（古い順）. 同一優先度内では先に起票したものが先に来る.
    3. **第三キー: `id`** の昇順（lexicographic 比較）. UUIDv4 同士の比較で同値はほぼ起きないが, 万が一 `createdAt` が完全同値の場合の決定論性を担保する.
  - **`dueDate` は並びキーから外す**（今日ビューは today のみで構成されるため必要ない. 本決定により BL-001 暫定実装の第一キーが消える）.

加えて, 本機能スコープの構造担保:

- **NFR-013 担保のフィールド不変条件**: ソートキーに使う `priority` / `createdAt` / `id` は通常編集経路（PATCH name / dueDate）で書き換わらないこと. BL-001 で確認済み（`createdAt` 不変）.

### 非機能要件

- **NFR-001（単一ワークフロー強制）**: 今日ビューにビュー切替（カンバン / カレンダー / リスト切替）・保存フィルタ・ユーザー定義ソートを持たない. 並び順を変える設定 UI を提供しない.
- **NFR-013（並び順の予測可能性）**: 同じデータ状態を入力すれば, クライアントを再描画しても, サーバを再起動しても, 別端末から見ても, 並び順は常に同一であること（決定論性）. 上記 FR-011 の 3 段キーで完全順序が確定する.

## 受け入れ基準

> 書き方は [`../../quality/acceptance-criteria.md`](../../quality/acceptance-criteria.md) に従う. Gherkin の Given/When/Then で表現する.

### 入口（FR-010）

```
シナリオ: Web クライアントを起動するとルート (/) に今日ビューが表示される
  Given Web クライアントの初期 URL は "/" である
  When  ユーザーがアプリを起動する
  Then  今日ビュー (TodayView) が描画される
  And   見出し「今日」が表示されている
```

```
シナリオ: 今日ビュー以外を入口に変更する設定は存在しない
  Given Web クライアントが起動している
  When  ユーザーが設定 UI（存在する場合）を確認する
  Then  「起動時の画面」「既定ビュー」を変える設定は提供されていない
```

### 表示対象の絞り込み（FR-010）

```
シナリオ: 今日ビューには dueDate = "today" のタスクのみが表示される
  Given タスク T_today { dueDate: "today", trashedAt: null } が存在する
  And   タスク T_tomorrow { dueDate: "tomorrow", trashedAt: null } が存在する
  When  今日ビューを表示する
  Then  一覧には T_today が含まれる
  And   一覧には T_tomorrow が含まれない
```

```
シナリオ: 今日ビューにはゴミ箱状態のタスクは表示されない
  Given タスク T_active { dueDate: "today", trashedAt: null } が存在する
  And   タスク T_trashed { dueDate: "today", trashedAt: <非 null>, trashedReason: "deleted" } が存在する
  And   タスク T_completed { dueDate: "today", trashedAt: <非 null>, trashedReason: "completed" } が存在する
  When  今日ビューを表示する
  Then  一覧には T_active のみが含まれる
  And   T_trashed および T_completed は含まれない
```

```
シナリオ: 今日ビューは全プロジェクト横断 + プロジェクト外 + ルーティン由来を含む
  Given タスク T1 { dueDate: "today", projectId: "P1", origin: "manual" } が存在する
  And   タスク T2 { dueDate: "today", projectId: "P2", origin: "manual" } が存在する
  And   タスク T3 { dueDate: "today", projectId: null,  origin: "manual" } が存在する
  And   タスク T4 { dueDate: "today", projectId: null,  origin: "routine" } が存在する
  When  今日ビューを表示する
  Then  T1, T2, T3, T4 のすべてが含まれる
```

### 並び順の本仕様（FR-011 / NFR-013）

```
シナリオ: 今日ビューは優先度の昇順（highest → normal → later）で並ぶ
  Given タスク A { priority: "later",   createdAt: "2026-06-08T08:00:00.000Z" }
  And   タスク B { priority: "normal",  createdAt: "2026-06-08T08:00:00.000Z" }
  And   タスク C { priority: "highest", createdAt: "2026-06-08T08:00:00.000Z" }
  And   いずれも dueDate = "today", trashedAt = null
  When  今日ビューを表示する
  Then  並び順は C, B, A になる
```

```
シナリオ: 同一優先度内では createdAt 昇順（古い順）で並ぶ
  Given タスク A { priority: "normal", createdAt: "2026-06-08T08:00:00.000Z" }
  And   タスク B { priority: "normal", createdAt: "2026-06-08T08:00:01.000Z" }
  And   タスク C { priority: "normal", createdAt: "2026-06-08T08:00:02.000Z" }
  When  今日ビューを表示する
  Then  並び順は A, B, C になる
```

```
シナリオ: 優先度と createdAt が同じなら id の昇順で並ぶ（決定論性の最終担保）
  Given タスク X { id: "00000000-0000-4000-8000-000000000002", priority: "normal", createdAt: "2026-06-08T08:00:00.000Z" }
  And   タスク Y { id: "00000000-0000-4000-8000-000000000001", priority: "normal", createdAt: "2026-06-08T08:00:00.000Z" }
  When  今日ビューを表示する
  Then  並び順は Y, X になる
```

```
シナリオ: 同じデータ状態であれば何度取得しても並び順は同じ（決定論性）
  Given タスク A, B, C, D が今日ビューの対象として存在する
  When  今日ビューを 2 回連続で取得する
  Then  2 回の取得結果は同じ並び順である
```

### "次の 1 つ" の一意化（FR-011）

```
シナリオ: 今日ビューの並びの先頭要素が「次の 1 つ」として参照可能である
  Given タスク A { priority: "highest" } と B { priority: "normal" } が今日ビューに含まれる
  When  今日ビューを取得する
  Then  レスポンスから「並び順の先頭タスク」を 1 つ取り出すことができ, それは A である
  ※ 「現在のタスク」(FocusSelection) はここでは未選択でよい. 本シナリオは「FocusSelection が未選択でも次の 1 つが一意に決まる」ことを担保する.
```

```
シナリオ: 今日のタスクが 0 件のとき「次の 1 つ」は存在しない
  Given dueDate = "today" かつ trashedAt = null のタスクが 1 件も存在しない
  When  今日ビューを取得する
  Then  一覧は空であり, 「次の 1 つ」も存在しない（取得側はその不在を識別できる）
```

```
シナリオ: 並びの先頭タスクを完了すると, 次に先頭になるのは 2 番目だったタスク
  Given タスク A { priority: "highest" }, B { priority: "highest", A より新しい createdAt }, C { priority: "normal" } が今日ビューに含まれる
  And   並び順は A, B, C である
  When  A を完了する（BL-003 の完了アクション）
  Then  今日ビューを再取得すると並び順は B, C になる
  And   「次の 1 つ」は B である
  ※ FocusSelection の自動繰り上げは BL-006 の責務. 本シナリオは「今日ビューの取得結果として B が先頭になる」ことのみ担保する.
```

### tomorrow タスクの扱い

```
シナリオ: 今日ビュー上で dueDate を tomorrow に切り替えるとそのタスクは今日ビューから消える
  Given 今日ビューに タスク T { dueDate: "today", version: 1 } が表示されている
  When  T の期限切替トグルを操作し dueDate を tomorrow に変更する（既存 BL-001 経路）
  Then  PATCH /api/v1/tasks/<T の id> が { dueDate: "tomorrow" } で送られ, 成功する
  And   今日ビューを再取得すると T は含まれない
```

```
シナリオ: tomorrow タスクは今日ビューには現れないが, データとしては存続している
  Given タスク T が { dueDate: "tomorrow", trashedAt: null } で存在する
  When  今日ビューを取得する
  Then  T は今日ビューに含まれない
  And   GET /api/v1/tasks?trashed=false で T が取得できる
  ※ tomorrow タスクの専用閲覧ビューは本 feature では提供しない（非ゴール）.
```

### 既存実装との整合（暫定 → 本実装の差し替え担保）

```
シナリオ: BL-001 暫定実装の「今日ビューに tomorrow タスクが現れる」挙動は本実装に置き換わる
  Given タスク T1 { dueDate: "today" } と T2 { dueDate: "tomorrow" } が存在する
  When  今日ビューを表示する
  Then  T1 のみが表示される
  And   既存テスト（web/__tests__/today-view.test.tsx 等）も today のみが対象となるよう更新されている
```

```
シナリオ: 既存タスク CRUD・優先度操作・完了操作は本機能の影響を受けず引き続き動作する
  Given BL-001 / BL-002 / BL-003 で導入された API (POST / PATCH / DELETE / POST :id/complete) が green であること
  When  本機能の今日ビュー API / UI を導入する
  Then  既存のテストが引き続き green である（並び順仕様の変更を除く）
```

### 並び順を変えるカスタマイズが存在しないこと（NFR-001）

```
シナリオ: 今日ビューに並び替えオプション UI / API パラメータは存在しない
  Given 今日ビュー API および UI が本機能で実装されている
  When  並び替えのオプションを探す
  Then  ソートキー切替・昇順降順切替などの UI コントロールは存在しない
  And   /today エンドポイントに並び順を変えるクエリパラメータは存在しない
```

## 未決事項 / 確認待ち

- **U-001 `/today` 専用エンドポイントを設けるか, `/tasks?dueDate=today` で済ますか**: API の表現方針. **保守側デフォルト案: `/api/v1/today` を新設する**（FR-010 / FR-011 / FR-013 が結びついた "今日ビュー" を API 上でも 1 級概念として表現するため. openapi.yaml の骨格・タグも既に `/today` を前提とした分離になっている. 詳細は plan.md §「重要な決定」で確定）.
- **U-002 並び順 3 段目に `id` を採用してよいか**: NFR-013（決定論性）を担保するため何らかの完全順序キーが必要. **保守側デフォルト案: `id` 昇順（UUIDv4 の lexicographic 比較）**. ユーザー視認順との直結性はないが, 同一 `createdAt` でのみ機能するタイブレーカーであり利用者からは事実上不可視. 別案として `updatedAt` 昇順や `name` 昇順もあるが, 編集で変動するためソートキーには不向き（NFR-013 リスク）.
- **U-003 並び順の本仕様で `dueDate` を捨てて良いか**: 今日ビューは today のみを対象とするため第一キーから `dueDate` を外した. 一覧 API `GET /api/v1/tasks`（trash 含む等）のサーバ側ソート規則は本 feature の責務ではないが, 矛盾を避けるため **`/api/v1/tasks` 側のサーバソートは「priority → createdAt → id」に統一する**（暫定 3 段ソートの第一キー `dueDate` を恒久的に削る）. ユーザー確認を要する.
- **U-004 「次の 1 つ」をレスポンス形状でどう表現するか**: 3 案あり.
  - 案 (a): `{ tasks: [...] }` のみを返し, 「先頭要素 = 次の 1 つ」という暗黙の解釈を契約とする（追加プロパティなし. 最小実装）.
  - 案 (b): `{ tasks: [...], nextTaskId: "..." | null }` のように先頭の id を明示プロパティとして返す.
  - 案 (c): `{ tasks: [...], next: <Task> | null }` のように先頭の Task オブジェクト全体を返す.
  - **保守側デフォルト案: 案 (b)**（クライアントが「先頭か否か」を id 比較で判定でき, 配列順序の偶発変更に強い. BL-006 FocusSelection が未選択時の既定値計算にも利用できる. Task 全体を二重に持つ案 (c) は冗長）.
- **U-005 ルーティン由来タスクと通常タスクで並び順を分けるべきか**: project.md UC-001 / UC-004 を読む限り両者を同列に並べる前提と読める. **保守側デフォルト案: 区別しない**（origin による分岐なし）. 確認のみ要する.
- **U-006 今日ビュー API のクエリパラメータ**: 並び順を変える設定は出さない（NFR-001）. ただし `trashed` / `dueDate` / `priority` 等のフィルタを今日ビュー上で提供するかは検討余地あり. **保守側デフォルト案: クエリパラメータなし（パラメータなしで取得 → 上記固定の絞り込み・並びを返す）**. NFR-001 と整合する最小構成. 復元はゴミ箱（BL-011）から行う前提.
- **U-007 今日ビューの再フェッチタイミング**: 完了・削除・期限切替・優先度変更の各操作後にクライアントが今日ビューを再取得すべきタイミング. **保守側デフォルト案: 各書き込み mutation 成功時に TanStack Query の invalidate で再取得**（楽観 UI の打ち消しを最小にするため）. 詳細は plan.md で確定し, 必要なら BL-014（Web クライアント基盤）の責務へ寄せる.
