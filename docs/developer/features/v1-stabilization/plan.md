# 設計・実装計画: v1.0.0 安定化

> [`spec.md`](spec.md) の要件を、どう実現するかに落とす。

## 方針概要

本機能は新たなコード実装を伴わない。既存の実装・テスト・バックログを照合し、
公開可能な状態であることを文書として証明する「最終監査と記録」が主な作業である。
成果物はすべて `docs/developer/features/v1-stabilization/` 配下に置く。

## 影響範囲

| 領域 | 変更内容 |
| --- | --- |
| API | なし |
| DB | なし |
| モジュール | なし |
| UI | なし |
| ドキュメント | `coverage-audit.md`・`test-results.md`・`release-checklist.md` を新規作成 |

## 設計詳細

### 作成するドキュメント

#### 1. `coverage-audit.md`（要件カバレッジ監査）

FR・NFR の全件を機械可読な表として記録する。表の列は次のとおり。

| 列 | 内容 |
| --- | --- |
| ID | FR-XXX / NFR-XXX |
| 要件概要 | project.md または仕様書からの引用 |
| 対応 BL | 実装が完了している BL 番号（複数可） |
| 対応テスト | 代表的なテストファイル・テスト名 |
| 適合状況 | `適合` / `対象外`（`非適合` は存在してはならない） |
| 備考 | 補足説明が必要な場合のみ |

バックログ整合性の確認セクションも同ファイルに設ける。
BL-001〜BL-023 を表形式で列挙し、状態（Done）を記録する。

#### 2. `test-results.md`（テスト通過記録）

`pnpm test`（またはプロジェクトで使用しているテスト実行コマンド）の実行結果を記録する。
記録項目:

- 実行日時（ISO 8601 形式）
- 実行コマンド
- 合計テスト件数
- 通過件数 / 失敗件数
- 実行環境（OS・Node.js バージョン等）
- テストランナーの出力サマリー（貼り付け）

#### 3. `release-checklist.md`（リリースチェックリスト）

次の 3 セクションで構成する。

**セクション 1: バックログ最終確認**
BL-001〜BL-024 の状態を一覧表示し、BL-024 以外が全て Done であることを確認するチェックボックス。

**セクション 2: リポジトリ公開準備**
BL-022 で整備済みの各ファイルの存在確認チェックボックス。
- `README.md`
- `LICENSE`
- `CONTRIBUTING.md`
- `docs/developer/features/oss-release-prep/` の完了確認

**セクション 3: テスト・監査**
- 全テスト green（`test-results.md` 参照）
- `coverage-audit.md` で FR・NFR 全件適合確認済み
- auditor による最終レビュー承認

**セクション 4: ドッグフーディング確認（手動）**
SC-001「1 か月以上日常的に使い続けられる」に対応する手動確認項目。
自動化できない性質のため、チェックボックス形式で定義する。

- 今日ビューを毎日開いてタスクを確認できている
- タスクの起票・完了・削除を日常的に行えている
- 日次リセットが問題なく動作している（カウントリセット・未完了繰越）
- プロジェクト機能を実際の作業文脈で使えている
- ルーティン機能が指定曜日に正しく生成されている
- フォーカス機能（現在のタスク）の自動繰り上げが期待通り動作している
- ゴミ箱・復元が問題なく使えている
- 1 か月以上継続して利用している（開始日と確認日を記録する）

**セクション 5: リリースゲート**
以下の条件をすべて満たしたときに `v1.0.0` タグを打つ。

| 条件 | 確認方法 |
| --- | --- |
| 全テスト green | `test-results.md` の失敗件数 = 0 |
| FR 全件適合 | `coverage-audit.md` に「非適合」行なし |
| NFR 全件適合 | `coverage-audit.md` に「非適合」行なし |
| BL-001〜BL-023 全件 Done | バックログ一覧 |
| リポジトリ公開準備完了 | `release-checklist.md` セクション 2 全チェック済み |
| ドッグフーディング確認 | `release-checklist.md` セクション 4 全チェック済み |
| auditor 承認 | auditor の最終レビュー Pass |

### 処理フロー（作業の順序）

1. プロジェクト憲章（`project.md`）から FR・NFR の全件リストを抽出する。
2. 各 FR・NFR に対し、対応する BL とテストを紐づけて `coverage-audit.md` を作成する。
3. テストを実行し、結果を `test-results.md` に記録する。
4. `release-checklist.md` を作成し、セクション 1〜5 を埋める。
5. セクション 1〜3・5 のチェックが完了したら auditor にレビューを依頼する。
6. auditor 承認後、セクション 4（ドッグフーディング）を手動で確認・記録する。
7. 全条件を満たしたら `v1.0.0` タグを打つ。

### 例外 / エラー処理

- 監査中に「非適合」が見つかった場合は、該当する BL を差し戻し修正する。修正後に再監査を行う。
- テスト失敗が見つかった場合は、`test-results.md` の記録を行わず、失敗の原因を修正してから再実行する。

## 重要な決定

- ドッグフーディング確認は自動化しない。期間計測や強制ロックは設けず、担当者の裁量で確認する。
- `v1.0.0` タグはリリースゲートの全条件を満たすまで打たない。条件を一部でも未完了の場合はタグを保留する。
- 既存の実装・テストコードは本機能の作業では変更しない（監査で問題が見つかった場合は別 BL として扱う）。

## リスク / 代替案

- リスク: FR・NFR の番号体系が project.md に網羅されていない場合、監査の基準が曖昧になる。
  - 対策: project.md の記述を正典とし、記述のない番号は「対象外」として備考に理由を記す。
- リスク: ドッグフーディング確認が主観的になる。
  - 対策: チェック項目を具体的な操作単位に分解し、期間の開始日・確認日を明記する運用とする。

## テスト方針

本機能自体に自動テストは不要（ドキュメント作成が主体のため）。
ただし次の点を auditor がレビュー時に確認する。

- `coverage-audit.md` に「非適合」行が存在しないこと。
- `test-results.md` の失敗件数が 0 であること。
- `release-checklist.md` の全チェックボックスが埋まっていること（ドッグフーディングを除く）。
