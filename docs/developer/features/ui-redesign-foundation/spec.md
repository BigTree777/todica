# 仕様: UI 全面刷新の総合仕様 (デザイン基盤と差分カタログの確定)

- 状態: ドラフト
- 関連: [`../../planning/backlog.md`](../../planning/backlog.md) BL-035 (および後続 BL-036 〜 BL-047 の前提)
- 由来要件: FR-003 (優先度の付与) / FR-004 (優先度の変更) / FR-010 (今日ビュー = アプリの入り口) / FR-011 (並び順) / FR-012 (現在のタスク) / FR-013 (現在のタスクの自動解除) / FR-014 (今日 → 明日) / FR-020 (プロジェクト管理) / FR-040 (完了数カウント) / FR-060 (ゴミ箱) / FR-061 (復元)
- 関連 NFR: NFR-001 (単一ワークフロー強制) / NFR-010 (最小手数の起票) / NFR-011 (フォーカス時の単独大表示) / NFR-012 (設定項目最小化) / NFR-013 (並び順の予測可能性)
- スコープ位置付け: 本 BL は **「UI 刷新ロードマップの総合仕様 (foundation)」** であり, 実装は行わない. 後続 BL-036 〜 BL-047 がそれぞれ独立して着手できる前提を確定する.

## 背景 / 課題

todica v1.0.0 までで FR / NFR の全要件は実装済みだが (BL-001 〜 BL-024 Done), 内部品質 / E2E 補強 (BL-025 〜 BL-034) を経て UI の構造的な負債が顕在化した. ユーザー (本人 / OSS 公開後の利用者) が **「今やる 1 つに集中する」** という project.md の中核体験を阻害する現状の問題点を以下に列挙する.

1. **`today-view.tsx` に「フォーカス」「起票」「一覧」「編集ダイアログ」「完了数」「優先度切替」「期限切替」「現在に設定 / 解除」が全て同居している** (約 620 行, 14 useMutation / 6 種類のボタンが各タスクカードに並ぶ). NFR-011 「現在のタスクが大きく単独で表示される」と NFR-010 「最小手数の起票」が同じ画面に同居しており, 両立できていない.
2. **タスクカードのアクションが 6 つ並ぶ** (優先度切替 / 編集 / 明日へ / 完了 / 削除 / 現在に設定). 1 カード 1 タスクの視認性が悪く, 「次の 1 つに集中する」UX に反する.
3. **プロジェクト追加導線が遠い**. `/projects` (ProjectsView) への画面遷移を要する. 今日ビューでタスクを起票しているときに新しいプロジェクトを思いついても, 別画面に飛ぶ必要があるため起票のテンポが切れる (NFR-010 違反).
4. **「明日のタスク」を一覧で確認する手段がない**. 今日ビュー上で「明日へ」操作で押し出すのみで, 押し出した後はゴミ箱以外で参照できない. ユーザーは「明日の準備」ができない.
5. **設定 / ゴミ箱 / ルーティン / プロジェクト の補助ビューに統一されたナビゲーション (グローバルナビ) がない**. 各 view が独自に `<main>` を持つだけで, view 間の往復に URL 直打ち / ブラウザ戻るボタンを要する. 単一ワークフロー (NFR-001) の「一本道」が UI で表現されていない.
6. **タイポグラフィ / 余白 / 角丸 / カラーの統一基盤 (デザイントークン) がない**. 各 view が素の HTML 要素のままで, モックアップが要求する「大きな見出し」「広い余白」「丸い枠」のいずれも表現されていない.

## ゴール / 非ゴール

### ゴール

- **モックアップ (本 spec §「UI モックアップの構造」) の構造的達成**:
  - 左サイドバー + メイン領域の 2 ペイン構成.
  - 「現在のタスク」「今日のタスク」「明日のタスク」の 3 ビュー切替.
  - 各ビューが単一責務を持ち (focus / today / tomorrow), 操作の混在を解消する.
- **後続 BL-036 〜 BL-047 の独立着手可能性の確保**:
  - 各 BL の境界 (担当 view / 変更ファイル / 仕様の範囲) を本 spec の「差分カタログ」で確定する.
  - 各 BL が他 BL の完了を待たずに着手でき, 並行開発が可能であること (依存関係グラフは plan.md で確定).
- **デザイン基盤の確定**:
  - タイポグラフィ規約 (見出し階層 H1/H2, 本文サイズ) を決める.
  - 余白 / 角丸 / カラーのトークン体系を決める (具体値の決定は BL-046 に委ねるが, トークン**体系**を本 BL で決める).
  - タスクカードのアクション最大数規約 (3 つまで) を決める.
  - 操作系の規約 (タップで選択 / トグルで切替 / select は使わない 等) を決める.

### 非ゴール

- **既存機能 (BL-001 〜 BL-024) の振る舞いを変えること**. CRUD / 完了 / 期限切替 / 優先度 / focus / リセット / ゴミ箱 / プロジェクト / ルーティン / カウンタ の各機能の **入出力・状態遷移・サーバ API・データモデル** は一切変更しない. 本 BL は UI レイヤの再構築のみを扱う.
- **データモデル / API / ドメイン層の変更**. サーバ側 (`server/src/`) と `domain/` パッケージは無改修で完結することを前提とする. API の追加・変更が必要な場合は対応する後続 BL の責務とし, 本 BL では「無改修で済む UI 再構築」のみを foundation とする.
- **Capacitor / Android 固有の UI 調整 (ドロワー / ハードウェアバックボタン等)**. Web の刷新が完了した後の別 BL (本 BL 完了後に backlog に追加する想定) で扱う. ただし「将来モバイルでドロワー化する余地」は spec §「未決事項」に Open Question として残す.
- **Lighthouse / アクセシビリティ監査の数値目標**. BL-029 で WCAG 2.1 AA の axe 検査は導入済み. 刷新後の view も violations 0 を維持する (受け入れ基準に含める) が, スコア目標値は本 BL では設けない.
- **コードの実装** (本 BL は spec / plan / tasks の確定のみ). 実装は後続 BL-036 〜 BL-047 が担当する.

## UI モックアップの構造

`local/image.png` (手書きワイヤフレーム, リポジトリ外) を文章化した構造定義. 後続 BL はこの記述を正本として参照する.

### モックアップ上段: 「現在のタスク」ビュー (focus-view)

- 画面全体を占める単一の大きな角丸枠 (タスクなし時も枠は表示される).
- 上部に見出し「現在のタスク」.
- 中央に枠 (タスク 1 件分). 中央には水平線のみ描かれており, タスクが選択されていれば名称が中央に大きく表示される想定.
- 枠の下部左に「削除」ボタン, 下部右に「完了」ボタン. **アクションはこの 2 つのみ**.
- 編集 / 優先度切替 / 明日にする / 現在解除 のボタンは置かない.

### モックアップ下段: 「今日のタスク」ビュー (today-view 刷新後)

- 左に縦並びの細いサイドバー. 上から順に「現在のタスク」「今日のタスク」「明日のタスク」の 3 リンクが小さなテキストで並ぶ.
- 右側のメイン領域に以下が縦に並ぶ:
  - 一番上に「今日のタスク」の大きな見出し (H1 相当).
  - 見出しの右上に「+プロジェクトの追加」ボタン.
  - 起票フォーム (角丸の枠で囲まれた 1 ブロック):
    - 「プロジェクト名 (トグルで選択)」と注記された横長ボタン (プロジェクト選択トグル).
    - 「タスク名」と書かれた水平線つき入力欄.
    - 右上に「☆☆☆」と「↑タップで選択」のラベル (優先度の星評価).
    - 右下に「追加」ボタン.
  - タスクカード (角丸の枠で囲まれた 1 ブロックが 1 タスク):
    - 上部に「プロジェクト名」.
    - 中央に「タスク名」(大きめの文字, 水平線つき).
    - 下部に 3 ボタン: 左に「削除」, 中央に「明日にする」, 右に「完了」. **このカードのアクションは 3 つのみ**.

### モックアップの示唆 (構造の規約として確定)

- **タスクカードのアクションは最大 3 つまで** (削除 / 期限切替 / 完了).
- **focus-view のアクションは 2 つまで** (削除 / 完了).
- **起票フォームの入力は 4 要素まで** (プロジェクト / タスク名 / 優先度 / 追加ボタン). **期限セレクトは置かない** (ビュー文脈で dueDate が決まる).
- **優先度 UI は星 3 つの評価式** (☆3=最優先, ☆2=普通, ☆1=後回し のマッピングは BL-040 で確定).
- **プロジェクト選択はトグル UI** (`<select>` は使わない).
- **完了数カウンタはモックに表示されていない** (FR-040 は必須仕様のため別の配置を BL-047 で確定する).

## 要件

本 BL は「**何を確定するか**」の foundation であり, 各要件は後続 BL が参照する規約として記述する.

### 機能要件 (本 BL では仕様の確定のみ. 実装は後続 BL)

- **REQ-1 ナビゲーション構造**
  - 左サイドバー + メイン領域の 2 ペイン構成を採用する.
  - サイドバー上部 (プライマリナビ) に 3 リンク: 「現在のタスク」 (`/focus`) / 「今日のタスク」 (`/today`) / 「明日のタスク」 (`/tomorrow`).
  - サイドバー下部 (セカンダリナビ) に補助メニュー: 「プロジェクト」 (`/projects`) / 「ルーティン」 (`/routines`) / 「ゴミ箱」 (`/trash`) / 「設定」 (`/settings`).
  - 起動直後 (`/`) は `/today` にリダイレクトする (現状維持. FR-010 「今日ビュー = アプリの入り口」).
  - サイドバーは常時表示 (Web デスクトップ前提). モバイルでのドロワー化は Open Question として残す.
- **REQ-2 3 ビューの責務分離**
  - **focus-view (`/focus`)**: 「現在のタスク」1 件のみを大きく表示. アクションは「削除」「完了」の 2 つのみ. 起票フォームは置かない. (担当: BL-037)
  - **today-view (`/today`, 刷新後)**: `dueDate = "today"` のタスク一覧 + 起票フォーム + プロジェクト追加ボタン. アクションは 3 つ (削除 / 明日にする / 完了). (担当: BL-039〜BL-042, BL-044)
  - **tomorrow-view (`/tomorrow`)**: `dueDate = "tomorrow"` のタスク一覧 + 起票フォーム (dueDate = tomorrow 固定). アクションは 3 つ (削除 / 今日にする / 完了). (担当: BL-038)
- **REQ-3 タスクカードのアクション最大数 = 3**
  - 1 タスクカード上に置けるアクションボタンは最大 3 つまで.
  - today-view では「削除 / 明日にする / 完了」, tomorrow-view では「削除 / 今日にする / 完了」を採用する.
  - 名称編集 / 優先度切替 / focus 設定 はカード上のアクションには含めない (詳細は REQ-5).
- **REQ-4 起票フォームの入力 4 要素**
  - プロジェクト選択 (トグル UI, null も 1 状態として扱う) / タスク名 (テキスト入力) / 優先度 (星 3 つ) / 「追加」ボタン.
  - 期限セレクトは置かない (ビュー文脈で dueDate が決まる). focus-view には起票フォームを置かない.
- **REQ-5 副次操作の経路 (カード上に置かない操作の代替経路)**
  - **名称編集**: タスクカードのタップ (またはダブルタップ) でモーダル / インラインダイアログを開く. 詳細は BL-042 / 別 BL で確定. 本 BL では「カード上にボタンとしては置かない」ことのみ確定する.
  - **優先度切替**: タスクカード上に表示される星 3 つの UI 自体をタップで切り替える (BL-040 で確定). カード上に「優先度切替」専用ボタンは置かない.
  - **現在のタスクに設定 (set focus)**: 今日ビューのタスクカードのジェスチャ (長押し or コンテキストメニュー or 専用ボタン) で行う. UX は BL-043 で確定. アクセシビリティ要件 (NFR-010) のためキーボード経路も必須.
  - **focus 解除**: focus-view 上の専用ボタンは置かず, 完了 / 削除時の自動繰上げ (BL-006 ロジック) で行う.
- **REQ-6 プロジェクト追加の今日ビュー上配置**
  - 今日ビューの見出し右上に「+プロジェクトの追加」ボタンを置く. クリックでモーダル (またはインラインフォーム) を開き, プロジェクト名 1 件で新規作成. (担当: BL-044)
  - 名称変更 / 削除は `/projects` (ProjectsView) で行う (本 BL では現状維持).
- **REQ-7 タイポグラフィ / 余白 / 角丸 / カラーのトークン体系**
  - **トークン体系の定義** (具体値は BL-046 で確定):
    - タイポグラフィ: `--font-size-h1` (大見出し) / `--font-size-h2` (節見出し) / `--font-size-body` / `--font-size-small`.
    - 余白: `--space-xs` / `--space-sm` / `--space-md` / `--space-lg` / `--space-xl` の 5 段階.
    - 角丸: `--radius-sm` / `--radius-md` / `--radius-lg` の 3 段階 (カード = lg を想定).
    - カラー: `--color-bg` / `--color-fg` / `--color-border` / `--color-accent` / `--color-danger`. ダークモード対応の有無は Open Question.
  - **トークンの配置場所**: `web/src/styles/tokens.css` を想定 (BL-046 で確定). CSS フレームワーク選定 (Tailwind / vanilla CSS / CSS Modules) は ADR で確定する.
- **REQ-8 完了数カウンタの配置**
  - FR-040 「今日の完了タスク数の表示」は必須仕様のため非表示化はできない.
  - モックには対応する要素がないため, 配置を本 BL では「未決」とし, BL-047 で確定する.
  - 候補: (a) サイドバー下部に小さく出す, (b) focus-view 内に統合する, (c) today-view の見出し右に小さく出す.

### 非機能要件

- **NFR-001 (単一ワークフロー強制) との整合**:
  - 並び順を変える UI / API パラメータを増やさない (現状維持).
  - ビュー切替 (カンバン / カレンダー等) を導入しない. 「現在 / 今日 / 明日」の 3 ビューはあくまで dueDate と focus の状態に対応する画面分解であり, **ユーザーが選ぶカスタマイズ視点ではない**.
- **NFR-010 (最小手数の起票) との整合**:
  - 起票フォームの必須入力は引き続き「タスク名」のみ (プロジェクト / 優先度はトグル UI の初期状態が既定値となる).
  - プロジェクト追加が画面遷移なしで行えるようになる (現状改善).
- **NFR-011 (フォーカス時の単独大表示) との整合**:
  - focus-view 単独で「現在のタスク」が画面全体を占める. 他タスクは表示しない.
- **NFR-012 (設定項目最小化) との整合**:
  - UI 刷新に伴う新規設定項目 (テーマ切替 / ビューレイアウト切替) を導入しない.
- **NFR-013 (並び順の予測可能性) との整合**:
  - 並び順は引き続きサーバ側 `priority → createdAt → id` の決定論的ソート. UI 側で再ソートしない.

## 未決事項 / 確認待ち (Open Questions)

- **U-001 モバイル (狭幅) でのサイドバーの扱い**: 常時表示のサイドバーは Web デスクトップでは自然だが, モバイル (Android / 狭幅ブラウザ) でドロワー化するか, 横スクロールにするか, それとも常時表示のままで内容を縮めるかは本 BL では決めない. Capacitor 配下の挙動と合わせて別 BL で扱う. **保守側デフォルト案: Web は常時表示, モバイルは別 BL で再検討**.
- **U-002 ダークモード対応の有無**: NFR-012 (設定項目最小化) と整合させるなら導入しない. ただし OS のシステム設定に追従するだけ (ユーザー設定を増やさない) なら整合可能. **保守側デフォルト案: 本 BL では「対応しない」を確定し, OS 追従が必要になった時点で別 BL で扱う**.
- **U-003 完了数カウンタの配置**: モックに対応箇所がない. (a) サイドバー下部, (b) focus-view 内, (c) today-view 見出し右, のいずれかを BL-047 で確定する. **保守側デフォルト案: (c) today-view 見出しの右に小さく出す**.
- **U-004 「現在に設定」の UX**: 長押し / コンテキストメニュー / 専用ボタンのいずれを採るか. キーボード経路の確保も必要 (NFR-010). BL-043 で確定する. **保守側デフォルト案: タスクカードのタップで開くアクションシート (削除 / 明日にする / 完了 / 現在に設定 / 編集 の 5 項目) を採用. キーボードはアクションシートをポップオーバーで提供**.
- **U-005 編集ダイアログのトリガ**: タスクカードのタップ全体に編集を割り当てるか, それとも別ジェスチャか. U-004 のアクションシート案を採るなら, シート内に「編集」を含めれば良い. **保守側デフォルト案: アクションシート内に「編集」を含める (U-004 と合わせる)**.
- **U-006 CSS フレームワーク選定**: Tailwind / vanilla CSS / CSS Modules / CSS-in-JS のいずれを採るか. ADR で確定する (BL-046 内で ADR 起票を想定). **保守側デフォルト案: vanilla CSS + CSS variables のトークン体系** (依存追加を避けるため. 個人開発・OSS の前提と整合).
- **U-007 プロジェクト数が多い場合のトグル UI 挙動**: BL-041 で確定. 個人利用 (project.md の前提) ではプロジェクト数は少ない (1 桁) 想定だが, 横スクロール / 行折返し / "もっと見る" のどれを採るか. **保守側デフォルト案: 行折返し (flex-wrap)**.
- **U-008 ルーティン由来タスクの表示と「明日にする」の整合**: FR-033 によりルーティン由来タスクは「明日にする」操作の対象外. UI で明日ボタンを非表示にするか, disabled にするか. 現状 `today-view.tsx` は非表示にしている. **保守側デフォルト案: 非表示 (現状維持)**.
- **U-009 サイドバーのアクティブ表示**: 現在いるビューをサイドバー上で太字 / 背景色 / 縦線 のいずれで示すか. **保守側デフォルト案: 太字 + 左に縦のアクセントライン**.
- **U-010 既存 view (settings/trash/routines/projects) のレイアウト**: AppShell に取り込むのは確定 (BL-045) だが, 内部レイアウトの刷新範囲はどこまでか. **保守側デフォルト案: ヘッダ / 余白 / 角丸 / フォントだけトークンを当て, ボタン配置・CRUD UI は現状維持**.
