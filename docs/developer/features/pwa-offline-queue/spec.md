# 仕様: PWA 化 + オフライン書込キュー

- 状態: 確定
- 関連: BL-018

## 背景 / 課題

Web クライアントは現在、サーバへの接続が断絶すると一切操作できない。個人運用サーバは
SLA 100% を保証しないため、通信断中でも読み取り閲覧と書込操作の継続が必要である。
また NFR-031 の Android 配布要件（PWA/TWA）を満たすため、Web Manifest によるインストール
可能化が前提となっている。ADR-0008 でこれらの実現方針が確定済みであり、BL-018 で実装する。

## ゴール / 非ゴール

- ゴール:
  - ブラウザ / ホーム画面からアプリとしてインストールできる（PWA Manifest + Service Worker）
  - シェル（HTML / JS / CSS）を pre-cache し、サーバ停止中でもアプリを起動できる
  - TanStack Query を導入し、サーバ状態管理（キャッシュ・楽観 UI・リトライ）を一元化する
  - API レスポンスを stale-while-revalidate でキャッシュし、オフライン中も前回データを閲覧できる
  - オフライン中の書込操作を IndexedDB キューに保持し、オンライン復帰時に自動再送する
  - 再送による二重登録を Idempotency-Key で防止する（サーバ側は既実装済み）
  - オフライン中の変更とサーバの変更が衝突した場合（412 応答）、UI でユーザーに解決を促す

- 非ゴール:
  - サーバ側の新規 API 追加（Idempotency-Key / If-Match は既実装済み）
  - Android ネイティブアプリの実装（TWA ラップは別タスク）
  - マルチユーザー・競合の自動マージ（単一ユーザー前提のため手動解決で十分）
  - バックグラウンド同期の進捗をリアルタイムに通知するプッシュ通知
  - Safari（iOS）の Background Sync 未対応環境でのオフライン書込（オンライン復帰後に手動再送を促す UI でカバー）

## 要件

- 機能要件:
  - PWA-001: `vite-plugin-pwa` で Web Manifest（name / short_name / icons / display=standalone / start_url 等）を生成する
  - PWA-002: Service Worker が HTML / JS / CSS / フォント等のシェルを pre-cache する
  - PWA-003: Chrome / Edge / Android Chrome のアドレスバーに「インストール」ボタンが表示される
  - PWA-004: インストール後、独立ウィンドウ（standalone モード）でアプリが起動する
  - TQ-001: TanStack Query（`@tanstack/react-query`）を `web/` に導入し、`QueryClientProvider` をアプリルートに配置する
  - TQ-002: 各 View の `useState` + 直接 `fetch` による状態管理を `useQuery` / `useMutation` に置き換える
  - TQ-003: 書込 mutation 成功後に関連クエリを invalidate し、再フェッチを起動する
  - RC-001: Service Worker の Workbox stale-while-revalidate 戦略で GET `/api/v1/*` レスポンスをキャッシュする
  - RC-002: オフライン中に GET リクエストが発生した場合、キャッシュから前回データを返す
  - WQ-001: `idb` ライブラリを導入し、書込キュー用 IndexedDB ストアを定義する
  - WQ-002: 書込操作（POST / PATCH / PUT / DELETE）を実行する前に、リクエスト情報（URL / method / headers / body / idempotency-key）を IndexedDB キューに永続化する
  - WQ-003: オンライン時はキューへの記録後、即座にリクエストを送信し、成功したらキューから削除する
  - WQ-004: オフライン時はキューに記録するだけでリクエスト送信を試みず、UI に「オフライン中」を示す表示を出す
  - WQ-005: Background Sync API が利用可能な場合、Service Worker がオンライン復帰イベントを受けてキューを自動再送する
  - WQ-006: Background Sync API が利用不可の場合、アプリがフォアグラウンドに戻った際にキューを手動再送する（`online` イベントをトリガーとする）
  - WQ-007: 再送時は IndexedDB に保存した Idempotency-Key をそのまま使用し、新しいキーを生成しない
  - CR-001: 再送時にサーバから 412 Precondition Failed が返った場合、衝突解決ダイアログを表示する
  - CR-002: ダイアログには「サーバの値を採用（クライアントの変更を破棄）」と「クライアントの値で再送（強制上書き）」の 2 択を提示する
  - CR-003: 「クライアントの値で再送」を選択した場合、最新の `version` を If-Match に設定して再送する
  - CR-004: 「サーバの値を採用」を選択した場合、キューから該当エントリを削除し、UI を最新データで更新する
  - SW-001: 新しい Service Worker がインストールされた際、ユーザーに「アップデートがあります。再読み込みしますか？」を通知する

- 非機能要件:
  - NFR-031: PWA インストールにより Android の TWA 配布基盤として機能すること
  - NFR-SW-01: Lighthouse の PWA 監査スコアが 100 点（または「インストール可能」全項目パス）であること
  - NFR-SW-02: キューに積まれた書込は最大 7 日間 IndexedDB に保持し、7 日経過後は自動で破棄する
  - NFR-SW-03: 同一 Idempotency-Key の再送は最大 5 回までリトライし、5 回失敗時はキューから除外してエラーログを記録する

## 受け入れ基準

### フェーズ A: PWA 基盤

```
シナリオ: インストールボタンの表示
  Given  HTTPS またはローカルホストで Web アプリを開いている
  When   ページが完全に読み込まれた
  Then   Chrome のアドレスバーに「インストール」ボタン（または PWA インストールプロンプト）が表示される

シナリオ: インストール後の standalone 起動
  Given  PWA をホーム画面にインストール済み
  When   ホーム画面のアイコンをタップ / クリックする
  Then   ブラウザのアドレスバーなしの独立ウィンドウでアプリが起動する

シナリオ: シェルのオフライン起動
  Given  アプリを一度オンラインで開き、Service Worker が pre-cache を完了している
  When   ネットワークを切断してからページをリロードする
  Then   アプリのシェル（HTML / JS / CSS）が表示され、エラー画面にならない
```

### フェーズ B: TanStack Query 導入

```
シナリオ: 今日ビューのデータ取得（TanStack Query 経由）
  Given  ユーザーがオンラインで /today を開く
  When   コンポーネントがマウントされる
  Then   useQuery が repository.today() を呼び出してタスク一覧が表示される

シナリオ: ミューテーション後の再フェッチ
  Given  今日ビューでタスクが表示されている
  When   タスクを完了する
  Then   useMutation の onSuccess で today クエリが invalidate され、一覧が最新の状態に更新される
```

### フェーズ C: 読み取りキャッシュ

```
シナリオ: オフライン中の前回データ表示
  Given  /today を一度オンラインで開き、タスク一覧が表示されている
  And    Service Worker が API レスポンスをキャッシュ済み
  When   ネットワークを切断してページを再読み込みする
  Then   前回取得したタスク一覧が表示され、「オフライン中」の旨が UI に表示される

シナリオ: オンライン復帰後のデータ更新
  Given  オフライン中でキャッシュデータが表示されている
  When   ネットワークが復帰する
  Then   バックグラウンドで最新データがフェッチされ、UI が更新される
```

### フェーズ D: 書込キュー + Background Sync

```
シナリオ: オフライン中のタスク追加（キュー記録）
  Given  ネットワークが切断されている
  When   ユーザーが今日ビューでタスクを追加する
  Then   操作が IndexedDB キューに保存される
  And    UI にはタスクが追加されたように（楽観的に）表示される
  And    「オフライン中 - 復帰後に同期します」などの表示が出る

シナリオ: オンライン復帰時の自動再送
  Given  オフライン中にタスクを追加してキューに保存されている
  When   ネットワークが復帰する（Background Sync またはオンラインイベント）
  Then   キューに保存されたリクエストがサーバに送信される
  And    送信成功後、該当エントリがキューから削除される
  And    今日ビューが最新データで更新される

シナリオ: 再送による二重登録の防止
  Given  キューに保存されたリクエストの Idempotency-Key が X である
  When   ネットワーク不安定により同じリクエストが 2 回サーバに届く
  Then   サーバは 2 回目のリクエストを無視し、同じタスクが 2 つ作成されない
```

### フェーズ E: 競合解決 UI

```
シナリオ: オフライン中に変更したタスクが衝突
  Given  オフライン中にタスク A を編集してキューに保存した
  And    その間にサーバ側でタスク A の version が更新されていた
  When   オンライン復帰後にキューを再送し、サーバが 412 を返す
  Then   衝突解決ダイアログが表示される
  And   「サーバの値を採用」「クライアントの値で再送」の 2 択が提示される

シナリオ: サーバの値を採用
  Given  衝突解決ダイアログが表示されている
  When   「サーバの値を採用」を選択する
  Then   クライアントの変更が破棄される
  And    キューから該当エントリが削除される
  And    UI がサーバの最新データで更新される

シナリオ: クライアントの値で再送
  Given  衝突解決ダイアログが表示されている
  When   「クライアントの値で再送」を選択する
  Then   サーバから取得した最新 version を If-Match に設定してリクエストを再送する
  And    再送が成功した場合、キューから該当エントリが削除される
  And    UI が更新後のデータで更新される
```

### Service Worker 更新通知

```
シナリオ: 新バージョンの Service Worker 検出
  Given  アプリが起動中
  When   新しいバージョンの Service Worker がインストールされた
  Then   「アップデートがあります。再読み込みしますか？」の通知が表示される
  And    ユーザーが再読み込みを承認すると新しい Service Worker が有効になる
```

## 未決事項 / 確認待ち

- Background Sync API が利用できない iOS Safari での書込キューの挙動を、フォールバック（`online` イベント）だけでカバーするか、追加 UI（「同期する」ボタン）も設けるかは実装段階で判断する
- Service Worker のキャッシュ名・バージョン管理（Workbox の `revision` 戦略）の詳細は実装段階で確定する
- IndexedDB のデータベース名・バージョン番号・ストア名は実装段階で確定する（本仕様では論理名のみ定義）
- 7 日間保持 / 5 回リトライの閾値は要件として定義したが、実装段階でサーバログとの照合が必要
