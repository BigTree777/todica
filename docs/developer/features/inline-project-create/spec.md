## 背景 / 課題

ui-redesign-foundation (BL-035) が指摘したとおり, 現状プロジェクトの新規作成は `/projects` (ProjectsView) への画面遷移を要する. 今日ビューでタスクを起票している最中に新しいプロジェクトを思いついても, 別画面に飛ぶ必要があり起票のテンポが切れる (NFR-010 違反).

モックアップ (foundation §「モックアップ下段」) では今日ビューの見出し右上に「+プロジェクトの追加」ボタンが配置されており, 別画面に遷移せずプロジェクトを追加できる導線が要求されている (foundation REQ-6).

BL-041 (project-toggle-ui, Done) により起票フォームのプロジェクト入力は `<ProjectToggle />` (巡回トグル) になっており, プロジェクト一覧は TanStack Query の `["projects"]` キャッシュ (today / tomorrow で共有) から供給される. 本 BL では今日ビュー上にモーダルでの追加導線を新設し, 作成したプロジェクトをこのトグル UI に即時反映する.

サーバ側 (`POST /api/v1/projects`, BL-016) と Repository 層 (`ProjectRepository.create`, HTTP 実装) は完成済みで無改修. 欠けているのは今日ビュー上の UI 導線のみである.

## UX の決定

仕様策定にあたり次の論点を確定した (検討の詳細は plan.md §「重要な決定」).

| 論点 | 決定 |
| --- | --- |
| 追加 UI の形態 | **モーダルダイアログ** (backlog の想定どおり). 実装はネイティブ `<dialog>` 要素 + `showModal()` を採用する. フォーカストラップ / Escape で閉じる / 背面の不活性化 (inert) / フォーカス復帰がブラウザネイティブで得られ, 独自実装 (既存 `ConflictDialog` の `div role="dialog"` 方式) より少ないコードで WCAG 2.1 AA のモーダル要件を満たせるため. |
| ボタンの配置 | **今日ビュー (`/today`) のみ**. モックの正本 (foundation §「モックアップ下段」) が今日ビューにのみ配置しており, foundation REQ-2 でも today-view の構成要素として「プロジェクト追加ボタン」を挙げているのは today のみ. tomorrow ビューには置かない (新規プロジェクトは `["projects"]` キャッシュ共有により tomorrow のトグルにも反映されるため, 導線が today のみでも機能は損なわれない). |
| 追加成功時の挙動 | モーダルを閉じ, `["projects"]` を invalidate してトグル UI に即時反映する. さらに **起票フォームのトグルに新規プロジェクトを自動選択する** (オンライン成功時のみ). 「起票中に思いついたプロジェクトをその場で作って使う」のが本導線の動機 (NFR-010) であり, 自動選択により追加後のトグル巡回タップが 0 回になるため. |
| 重複名称 | **許容する** (チェックしない). project-crud (BL-016) で「同名プロジェクトを複数作成できる」「一意性保証は行わない」が確定済みであり, サーバ仕様と整合させる. |
| オフライン時 | 既存 ProjectsView の作成 mutation と同型で **offline-queue に乗せる** (楽観成功でモーダルは閉じる). ただしトグルへの即時反映と自動選択はオンライン成功時のみ保証する (詳細は REQ-8). |

## ゴール / 非ゴール

### ゴール

- 今日ビュー (`/today`) の見出し領域 (右上) に「＋プロジェクトの追加」ボタンを置き, クリックでプロジェクト追加モーダルが開く (foundation REQ-6 / FR-020 の今日ビュー上の導線).
- モーダルは名称 1 件のみを受け取り, `ProjectRepository.create()` (= `POST /api/v1/projects`) で作成する.
- 作成成功でモーダルが閉じ, 起票フォームの `<ProjectToggle />` に新規プロジェクトが即時反映される (BL-044 完了の目安).
- 作成成功時 (オンライン), 起票フォームのトグルが新規プロジェクトを自動選択した状態になり, そのまま起票すると `projectId` に新規プロジェクトの id が送信される.
- マウス / タッチ / キーボードのいずれでも「開く → 入力 → 作成 → 反映」が同等に完結する.
- モーダルはフォーカストラップ・Escape クローズ・閉鎖時のフォーカス復帰を備え, axe (WCAG 2.1 AA) violations 0 件を維持する.

### 非ゴール

- **`/projects` (ProjectsView) の変更**: 名称変更 / 削除を担う管理画面としてそのまま残す (backlog BL-044 で明示されたスコープ外). ProjectsView 内の作成フォームも撤去しない (導線の重複は許容. 整理するなら別 BL).
- **tomorrow ビュー / focus ビューへのボタン配置**: 置かない (上記 UX の決定. モック正本に従う).
- **サーバ API / ドメイン層 / `ProjectRepository` インターフェイスの変更**: `POST /api/v1/projects` (BL-016) を無改修で再利用する. バリデーション (`INVALID_PROJECT_NAME`: 空 / 201 文字以上 / 制御文字) はサーバ既存仕様のまま.
- **プロジェクトの名称変更 / 削除をモーダルから行うこと**: 追加 (create) 専用. 変更 / 削除は `/projects` の責務.
- **既存 `ConflictDialog` のネイティブ `<dialog>` への移行**: 本 BL は新規モーダルにのみ `<dialog>` を採用する. 既存ダイアログの移行は別 BL.
- **デザイントークン化 / 視覚スタイルの作り込み**: BL-046 の責務. 本 BL はコンポーネントローカルの最小 CSS (`::backdrop` 含む) で WCAG AA を満たすところまで. `/* TODO(BL-046) */` マーカーを残す.
- **重複名称の警告 / 抑止**: サーバ仕様 (同名許容) と整合させ, チェックしない.
- **入力値のトリム / 正規化**: 既存 ProjectsView の作成フォームと同じく行わない (サーバの `INVALID_PROJECT_NAME` 検証が正本).

## 要件

### 機能要件

- **REQ-1 (今日ビューの見出し領域に「＋プロジェクトの追加」ボタンを置く)**
  - `/today` の見出し (`<h1>今日</h1>`) と同じヘッダ領域に, `<button type="button">` を 1 つ置く.
  - 表示ラベルおよびアクセシブルネームは「＋プロジェクトの追加」とする (モックの文言「+プロジェクトの追加」に対応. 記号は全角「＋」で統一し, アクセシブルネームは可視ラベルと一致させる — WCAG 2.5.3 Label in Name).
  - DOM 上の位置は `<h1>` 直後 (起票フォームより前) とする. 「右上」の視覚配置はローカル CSS で最小限実現し, 仕上げは BL-046 に委ねる.
  - `/tomorrow` / `/focus` には置かない.

- **REQ-2 (追加 UI はモーダルダイアログ)**
  - ボタンの作動でモーダルが開く. 画面遷移 (URL 変更) は発生しない.
  - 実装はネイティブ `<dialog>` 要素を用い, `showModal()` で開く. これによりフォーカストラップ・背面の不活性化・Escape による閉鎖をブラウザネイティブで満たす.
  - ダイアログには見出し「プロジェクトの追加」を置き, `aria-labelledby` でダイアログのアクセシブルネームとする.
  - 開いた直後, フォーカスは名称入力欄にある.
  - 閉じたとき, フォーカスは開いたボタン (「＋プロジェクトの追加」) に戻る (`showModal()` のネイティブ挙動).

- **REQ-3 (モーダルの入力はプロジェクト名 1 件のみ)**
  - フォーム構成: ラベル「プロジェクト名」付きのテキスト入力 1 つ + 送信ボタン「追加」 + 閉じるボタン「キャンセル」. これ以外の入力項目を置かない (NFR-012: 項目最小化).
  - 入力欄は `required` とし, `maxLength` を 200 とする (サーバの 1〜200 文字制約に整合するクライアント側ガード).
  - 「追加」は `type="submit"`, 「キャンセル」は `type="button"` とする (Enter キーで送信が動く).

- **REQ-4 (作成と即時反映)**
  - 「追加」の作動で `ProjectRepository.create({ id: <クライアント生成 UUID>, name })` を呼ぶ (= `POST /api/v1/projects`, Idempotency-Key 付与は既存 HttpProjectRepository / 既存 mutation 慣行のまま).
  - 成功時:
    1. モーダルを閉じ, 名称入力をクリアする.
    2. `["projects"]` query を invalidate する. today / tomorrow の `<ProjectToggle />` は同一 query キャッシュを参照しているため, 再フェッチ完了をもって両ビューのトグル巡回に新規プロジェクトが現れる (即時反映).
    3. オンライン成功時 (サーバから `Project` が返ったとき) は, 今日ビューの起票フォームのトグル値を新規プロジェクトの id に自動選択する. トグルのボタン表面に新規プロジェクト名が表示される.
  - 自動選択後の挙動は BL-041 既存仕様のまま: そのまま起票すると `CreateTaskCommand.projectId` に新規プロジェクトの id が送信され, 起票成功後にトグルは「（未分類）」へリセットされる (project-toggle-ui AC-10 互換).

- **REQ-5 (閉鎖経路とリセット)**
  - モーダルを閉じる経路は次の 3 つ: (a) 作成成功, (b) 「キャンセル」ボタン, (c) Escape キー (`<dialog>` ネイティブの cancel).
  - (b) / (c) で閉じた場合, `repository.create()` は呼ばれない (POST が飛ばない).
  - いずれの経路でも, 次回開いたとき名称入力は空である (作成失敗で開いたままのときを除き, 閉鎖時に入力状態を破棄する).
  - モーダルの背景 (backdrop) クリックでは閉じない (誤操作による入力消失を防ぐ. 閉鎖経路を上記 3 つに限定する).

- **REQ-6 (バリデーション)**
  - 空名称: クライアント側で `required` により送信を抑止する (POST は飛ばず, モーダルは開いたまま). サーバ側の `INVALID_PROJECT_NAME` (400) は最終ガードとして既存のまま.
  - 201 文字以上: `maxLength=200` により入力段階で抑止する.
  - 制御文字: クライアント側では検査しない (サーバの 400 に委ね, REQ-7 のエラー処理に流す).
  - 重複名称: 検査しない (サーバ仕様で同名許容. project-crud 確定済み).

- **REQ-7 (エラー処理と再試行可能性)**
  - 通信失敗・サーバエラー (400 / 401 / 5xx) 時は `notifyError("通信に失敗しました")` を表示する (BL-034 既存機構).
  - 失敗時, モーダルは**開いたまま**とし, 入力した名称を保持する (そのまま再試行できる).
  - 送信中 (mutation pending) は「追加」ボタンを disabled にし, 多重送信を防ぐ.
  - `ConflictDialog` は使わない (create は If-Match を持たず 412 が発生しないため).

- **REQ-8 (オフライン時の挙動)**
  - 既存 ProjectsView の作成 mutation と同型とする: offline-queue に `POST /api/v1/projects` エントリを enqueue し, `!navigator.onLine` のときは楽観成功として扱う (モーダルは閉じる). オンライン復帰時に BL-018 既存機構が flush する.
  - ただしオフライン時は `["projects"]` の再フェッチが成功しないため, トグルへの即時反映および自動選択 (REQ-4 の 2. / 3.) は**オンライン成功時のみ保証**する. オフラインで作成したプロジェクトは同期完了後の再フェッチでトグルに現れる.

- **REQ-9 (ProjectsView は無改修)**
  - `/projects` の一覧 / 作成フォーム / 名称変更 / 削除は現状維持. 今日ビューから作成したプロジェクトは, `/projects` を開けば一覧 (name 昇順) に表示される (同じサーバ正本を参照するため).

### 非機能要件

- **NFR-A11Y**: `e2e/a11y.spec.ts` の axe 検査 (WCAG 2.1 AA) で violations 0 件を維持する. **モーダルを開いた状態の `/today`** も axe スキャン対象に追加し violations 0 件とする.
- **NFR-KEYBOARD**: キーボードのみ (Tab / Enter / Escape) で「ボタン到達 → モーダルを開く → 名称入力 → 作成 → トグル反映」および「Escape で取消」が完結する (BL-029 以降のキーボード操作性慣行).
- **NFR-010 整合**: プロジェクト追加が画面遷移なしで行える (foundation §NFR-010 整合の現状改善項目). 起票フォームの入力 4 要素 (foundation REQ-4) は不変 — 本ボタンは起票フォームの外 (見出し領域) に置く.
- **NFR-COMPAT**: サーバ API / DB / ドメイン層 / `ProjectRepository` インターフェイス無改修. `POST /api/v1/projects` の body / ヘッダ形式は変えない.
- **NFR-CONSISTENCY**: mutation の組み立て (enqueue / offline 分岐 / notifyError) は ProjectsView `createMutation` (BL-016 / BL-018 / BL-034) と同型とし, 既存慣行から逸脱しない.

## 未決事項 / 確認待ち

- **U-1 (ボタンラベルの記号)**: 表示ラベルは「＋プロジェクトの追加」(全角＋) を採用する. screen reader が「＋」を読み上げる冗長さよりも, 可視ラベルとアクセシブルネームの一致 (WCAG 2.5.3) を優先した. 「＋」を `aria-hidden` の装飾に分離する案は実装が複雑になるため不採用. 異論があれば plan 確定前に変更する.
- **U-2 (ProjectsView 作成フォームとの導線重複)**: 本 BL 後, 作成導線が今日ビューのモーダルと `/projects` の 2 箇所になる. backlog の記載 (「ProjectsView は名称変更/削除を担う管理画面として残す」) を「作成フォームの撤去までは要求しない (無改修)」と解釈した. 作成フォーム撤去で管理画面を変更/削除専任にするかは別 BL でユーザーに確認する.
- **U-3 (自動選択の是非)**: REQ-4 で「オンライン成功時は新規プロジェクトを自動選択する」を確定した (NFR-010 / 本導線の動機との整合). 「反映のみで選択しない」案は, 追加後にトグルを巡回し直す手数が増えるため不採用. ただし「起票文脈でなく単に登録したかっただけ」の場合も選択状態になる副作用がある — 起票成功時のリセット (BL-041 AC-10) で自然に解消されるため許容した.
- **U-4 (オフライン E2E の検証範囲)**: REQ-8 で enqueue 方針を既存機構踏襲としたが, プロジェクト作成のオフライン E2E は**省略する** (BL-043 U-4 と同じ判断. キュー機構自体は BL-018 でテスト済みで, mutation の組み立ては ProjectsView createMutation と同型のため).
- **U-5 (jsdom での `<dialog>` 検証)**: jsdom は 25/26 とも `HTMLDialogElement.showModal` / `close` が未実装 (jsdom issue #3294. 策定時の「対応済み」は誤りで実装時に判明) のため, `web/__tests__/setup.ts` に open 属性の付け外し + close イベント発火のみの最小 polyfill を置いて単体テストで開閉と送信を検証する. フォーカストラップ / Escape のネイティブ挙動は jsdom では再現しないため, Playwright E2E (実ブラウザ) を正とする (AC-2 / AC-6 / AC-11 / AC-12).
- **U-6 (見出し文言)**: モックの見出しは「今日のタスク」だが現行 `<h1>` は「今日」. 見出し文言の変更は本 BL のスコープ外 (BL-046 以降の視覚仕上げ, または別 BL).
- **U-7 (today-view カード副情報のプロジェクト名表示 — 決定済み: AC-4 から除外)**: 初版 AC-4 に「タスクカードの副情報にプロジェクト名『仕事』が表示される」を含めていたが, **削除を確定**した. 根拠: (1) 現行 today-view のカードはプロジェクト名を表示しない (BL-041 の E2E でも「今日ビューのカード副情報は非ゴール」と確認済み. tomorrow-view は表示あり). (2) plan の「既存の起票フォーム / タスクカードには触らない」方針, および本 spec 冒頭の「カード / アクション規約に触らない」(task-card-actions / set-focus-gesture を正典) と矛盾していた. (3) BL-044 の本旨は「今日ビューからのプロジェクト追加導線」であり, カード副情報の表示は別関心事. projectId の送信自体は AC-4 の `POST /api/v1/tasks` body 検証で担保される. today-view カードへのプロジェクト名表示が必要なら別 BL として起票する.
